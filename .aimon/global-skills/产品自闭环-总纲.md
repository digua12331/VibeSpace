# 产品自闭环 · 总纲

把一个 GitHub issue 走完「捞取 → 改代码 → 录演示视频 → 发 PR + 发社交平台 → 关闭 issue」整条流水线。本 skill 是总纲，定义四步之间的数据约定；每一步的具体命令在对应的分步 skill 里（`捞需求-github-issues` / `本地实现-切分支改代码` / `录演示视频-hyperframes` / `发布-pr与社交平台`）。

## 四步顺序（不可乱序）

1. **捞需求** — 从 GitHub 拉 open 的 enhancement issue，挑一个高优的
2. **本地实现** — 切分支、改代码、本地测试通过
3. **录演示视频** — 把新功能做成一段 MP4
4. **发布** — 提 PR、（人工确认后）发视频、评论并关闭 issue

## 数据契约（四步靠这套约定传递状态，不要自己另发明）

整条闭环围绕一个 issue 编号 `<N>` 展开：

- **issue 编号 `<N>`**：第 1 步选定后，后面三步全程沿用。
- **分支名**：固定 `feat-ai-issue-<N>`（第 2 步创建，第 4 步 push）。
- **视频产物路径**：固定 `<项目根>/.aimon/artifacts/issue-<N>/demo.mp4`（第 3 步产出，第 4 步发布读取）。
- **PR**：body 必须含一行 `Closes #<N>`，并写明演示视频。
- **一次闭环 = 一个任务**：建议任务名含 issue 编号（如「闭环修 issue #<N>」），方便回溯。

## 状态跟踪（复用 VibeSpace 既有设施，不要另造）

- 不要新建状态文件 / 数据库。进度就记在这次 VibeSpace 任务的 `tasks.md`（Dev Docs 三段式的任务清单），或在对话里逐步汇报。
- 每一步的关键操作（拉 issue、切分支、跑测试、render、push、发布）都要在回复里说清楚做了什么、结果如何 —— VibeSpace 的操作日志（LogsView）会留痕，便于事后回放。

## 前置依赖（缺了就停下告诉大哥，不要硬跑）

- `gh`（GitHub CLI，已 `gh auth login` 登录）
- `git`
- Node + npx + FFmpeg（录视频用）
- Python 3 + 大哥自备的 `publish_social.py`（发社交平台用）

## 红线

第 4 步发布前必须人工确认，详见 `发布-pr与社交平台` skill —— AI 不准自己执行 push / PR / 发视频命令。
