# 团队派工：何时开 Task 工具开 subagent

VibeSpace 已经做了 s04 subagent run 卡片层（hook 抓 Task 调用挂到父 session），所以**主理人能看到你派出去的子工**。这影响"什么时候该派"的判断。

## 适合派 subagent 的场景

1. **多目录并行调研**：要看 packages/server/src/ + packages/web/src/ + scripts/ 三个独立部分时，三个 Explore subagent 并行比串行快很多
2. **先调研后实施**：不熟的代码区域先派一个 Explore 子工产清单，再用主上下文综合
3. **大文件遍历搜索**：要在 200+ 文件里找某模式，派 general-purpose 子工去搜，避免 grep 后逐个 Read 占父上下文
4. **独立可校验的子任务**：比如"调研每个 routes/*.ts 是否走了 serverLog 起止配对"——子工出列表父 agent 再决策

## 不适合派的场景

1. **核心决策**：plan / context 阶段的设计判断不要外包——主 agent 才有完整上下文判断"为什么这个方案而不是那个"
2. **流式编辑**：连续改 5 个文件（每个改动依赖前一个的结果）用主线性 Edit 比派 subagent 强；subagent 写文件主 agent 看不到中间状态
3. **debug + 修复**：bug 排查需要主 agent 持续追踪假设链，派 subagent 等于断链
4. **简单 grep**：单个关键词的 Grep 工具直接调，比派 subagent 包装 Bash grep 快 10 倍

## subagent_type 怎么选

### 项目级 `vibespace-*`（**优先级高于通用 type**——本仓库专用，跟 git 走）

定义在 `.claude/agents/`，跟项目语境深度耦合：

- **`vibespace-explorer`** — 项目代码地图测绘（read-only，返回 ≤30 行清单）
- **`vibespace-route-author`** — 加 fastify route 端到端切片
- **`vibespace-db-scribe`** — DB schema 三处套路加列
- **`vibespace-ui-decorator`** — EditorArea / SessionView badge / chip 装饰
- **`vibespace-smoke-author`** — scripts/*-smoke.mjs 端到端验收
- **`vibespace-browser-tester`** — 浏览器 V1..V5 自动验收（用 browser-use MCP）
- **`vibespace-rules-auditor`** — CLAUDE.md / .aimon/skills 合规审稿（read-only）

完整角色对照见 `dev/agent-team-blueprint.md`。**改 VibeSpace 仓库代码优先用这些**——它们 system prompt 里都点了项目特定 skill 文件。

### 通用 type（兜底——非本仓库特定的活，或 vibespace-* 没覆盖到的边缘场景）

- **`general-purpose`** — 默认通用调研 / 多步搜索，最稳
- **`Explore`** — 仅快速读文件 + 搜内容，不能写
- **`Plan`** — 写实施计划，read-only
- **`feature-dev:code-explorer`** — 给现有 feature 做深度代码地图（重量级，不要轻易上）
- **`feature-dev:code-reviewer`** — 审 plan 或代码改动符合 CLAUDE.md（拿不准方案对不对时用）
- **`codex:rescue`** — 卡 2-3 轮跑不通时换 Codex 视角；按 CLAUDE.md 熔断规则触发

不在以上两个清单里的 subagent_type 要先看 `.claude/agents/` 是否真有；瞎填会挂。

## prompt 写法

派 subagent 的 prompt 一定要**明确返回格式**，否则 subagent 会写一大坨进父上下文：

```
要求：
- 返回 ≤ 20 行总结
- 每行格式 "file:line - 一句话"
- 不要解释，不要复制代码，只列发现
```

不写返回约束等于自己挖坑——一个不带约束的 Explore subagent 经常返回 5KB 进父上下文，跟你"为了省上下文才派 subagent"的初衷相悖。

## 主理人能看见

派出去的子工会出现在 SessionView 顶部 violet chip bar，主理人可以点 chip 看 prompt + 状态。这意味着：

- **不要派"实验性"子工**（你也不知道它会跑啥的）—— chip 上一显示主理人会问"这是干嘛的"
- **prompt 第一行写清楚目标**—— chip 的 description 字段就是它，主理人扫一眼能懂的程度

## 不适用

- 单一 session 单线 vibe coding（一个人改一处代码）→ 不需要 subagent，主线干完
- 已经在 subagent 里再嵌套调 Task（v1 不支持层级，会挂在最外层 session 名下，混乱）
- "感觉派 subagent 比较 cool"——这是 over-engineering 的红灯
