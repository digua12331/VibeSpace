# 操作日志埋点（CLAUDE.md 强约束的具象化）

## 必埋的范围

- 新增 UI 操作（按钮、菜单项、对话框提交、拖放、快捷键）
- 新增 mutation API（写 DB、改文件、起/停进程、调外部 CLI、发网络请求）
- 修复影响用户行为的 bug（行为变了就埋）

## 豁免（不要强行加，否则规则变废纸）

typo / 重构 / 类型修复 / 内部 helper / 轮询 / 心跳 / WS 订阅确认 / 测试代码 / 启动脚本路径打印

## 起止配对（默认形态）

**一次操作至少两条日志**：

- 开始：`level=info` `msg='<action> 开始'`
- 终点：成功 `level=info msg='<action> 成功 (Nms)'`；失败 `level=error msg='<action> 失败: <reason>' meta.error = { name, message, stack }`

不要只记一条——只有开始没终点 = 不知道"卡住"还是"完成"；只有终点没开始 = 不知道耗时。

## 工具

- **前端**用 `logAction(scope, action, fn, ctx?)`（`packages/web/src/logs.ts`）包装 mutation。手动两次 `pushLog` 只在"这次日志跟任何异步无关"时用。
- **后端**用 `serverLog(level, scope, msg, extra?)`（`packages/server/src/log-bus.ts`）。**不要 `console.log`** 进业务路径——它不到 LogsView 也不落盘。
- **`app.log.warn` ≠ serverLog**：fastify 自带 logger 不进 LogsView。要让 LogsView 看到失败必须 `serverLog('error', ...)`。

## scope / action 命名

- `scope` 用小写单词：`project` / `session` / `docs` / `git` / `installer` / `server` / `subagent` / `skills` / `jobs` / `inbox` ...保持与 LogsView 既有过滤一致
- `action` 用动词：`create` / `start` / `archive` / `delete` / `bind-task` / `unbind-task`

## meta 必填字段

- 必填：`level` / `scope` / `msg`
- 能拿到就附：`projectId` / `sessionId` / `meta`
- `meta` 必须 **JSON-serializable**（`JSON.stringify` 不抛）且 **≤ 2KB**——不要把整份 diff、整棵 AST、DOM 节点塞进去

## 验收必跑

任何"新增 UI 操作"或"新增 mutation API"任务，对应步骤的 `verify` 必须显式写出：

- "在浏览器 LogsView 看到 `scope=X action=Y` 的起止配对"
- 失败分支至少有一次 ERROR 条目被人工触发验证过

没有这条 verify 视为步骤未完成。

## 现成模板（最近落地）

- 起止配对：`server/src/jobs-service.ts` 的 register/finish 双 serverLog
- 前端 logAction 包裹：`web/src/components/StartSessionMenu.tsx` 的 `start()` 函数
- ERROR 路径：`routes/sessions.ts` 的 `worktree-add 失败` / `routes/hooks.ts` 的 `subagent registerStart failed`

## 不适用

- 高频 IPC / WS 帧不要每次都写日志（会被淹掉）
- 内部纯函数 / 工具方法不需要埋点
- 跑了几年的稳定路径不要追加埋点（添加埋点本身有 bug 引入风险）
