# Vibe Coding 协作总览

> 面向主理人。你不需要懂代码，也不需要参与技术细节选择。
> 这份文档只回答一件事：以后怎么让 AI 少说多做，同时把事做得可验收、可追踪、可复盘。

## 一句话原则

你只管方向和验收；主 Claude 负责拆解、派工、执行、验证和留痕；Harness 负责隔离、日志、后台任务和记忆。

除非影响你最终看到的结果、已有数据安全，或验收方式不成立，否则 AI 不应该打断你。

## 你只需要做什么

| 你做什么 | 什么时候做 | 目的 |
|---|---|---|
| 提需求 | 任务开始 | 说清楚想要的结果即可，技术细节不用管 |
| 看 plan 摘要 | 三模型会审并写完 plan 后 | 确认大方向对不对 |
| 最终验收 | AI 交付后 | 按“去哪里点哪里”检查效果 |
| 点归档 | 验收通过后 | 让后台自动沉淀经验到记忆 |
| 偶尔写长期偏好 | 有稳定偏好时 | 追加到 `dev/memory/manual.md` |

你不需要做：

- 不需要给任务起名。
- 不需要确认 `context.md`。
- 不需要确认 `tasks.md`。
- 不需要挑技术方案、库、函数名、目录结构。
- 不需要手动勾选任务清单。
- 不需要看完整执行过程。

## AI 会自动做什么

一次正常任务的链路是：

```text
你提需求
  -> Claude / Gemini / Codex 三模型会审
  -> Codex 定稿 plan.md，主 Claude 白话化落盘
  -> 你只确认大方向
  -> 主 Claude 写 context.md
  -> 主 Claude 写 tasks.md / tasks.json
  -> 主 Claude 按任务类型派 subagent
  -> AI 改文件、跑验证、更新任务状态
  -> AI 给你短交付摘要
  -> 你验收并点归档
  -> Harness 后台沉淀经验到 memory
```

这里有三个角色：

| 角色 | 负责什么 | 类比 |
|---|---|---|
| Dev Docs | 记录方向、上下文、任务进度和验收标准 | 做事账本 |
| Team Agent | 决定谁来干活，主 Claude 统筹，subagent 执行 | 项目经理 + 专业工人 |
| Harness | 提供隔离、日志、hook、后台 job、记忆 | 施工护栏和档案系统 |

## 唯一需要你确认的地方

整条流程里你只点一次头：Plan 阶段。

Plan 阶段默认不是 Claude 单独想方案，而是 Claude 先读项目现状，Gemini 补边界，Codex 综合定稿，最后由主 Claude 把 plan 写成你看得懂的摘要。你不需要分别看三份方案，只看最终 plan 的主理人摘要。

你主要看 plan 最前面的 3-5 行“主理人摘要”：

```md
这次要做什么：
做完后在哪里点哪里能看到：
会不会动到已有数据/界面：
验收方式：
风险：
```

这段是给非程序员看的。**如果你看不懂，那就是 AI 没写合格**——直接回一句“摘要看不懂，重写”，不用客气。后面的 context、tasks、执行、验证都由 AI 自己推进，不再要你确认。

## AI 跟你说话的规矩

主理人不懂代码这件事是硬性约束。所有发给你的话（plan 摘要、交付总结、错误说明、回复消息）必须遵守：

- **专业术语第一次出现要括号翻译**。比如：“worktree（git 的临时副本，几个并行任务互不踩脚）”“mutation（会改数据库或文件的操作）”“payload（一次请求里夹带的数据）”。**裸丢术语就是没遵守**。
- **翻译成你看得见的变化**。不是“加了 zod 校验”，而是“前端发非法数据时后端会拦下来并提示”；不是“重构了 sessions 路由”，而是“会话列表打开速度变快了，行为不变”。
- **让你拍板的永远是“要不要这样做 / 在哪里能验收”**，不是“用 A 还是 B”。后者是 AI 自己的活，拿来问你说明它越界了。

如果某次 AI 给你的话不符合上面三条，可以直接打回去：“不懂，换人话”。

## AI 必须问你的情况

AI 只有遇到下面情况才应该停下来问你：

| 情况 | 为什么必须问 |
|---|---|
| 大方向有两种解读 | 你这句话能理解成 A 也能理解成 B，且**会让你最终看到的东西不一样**（界面不同 / 数据流向不同 / 能否撤销 / 步骤变没变） |
| 会动已有数据或不可逆操作 | 例如删除、迁移、覆盖、批量修改 |
| 验收方式不成立 | 例如原本说能在浏览器看到，但实际没有入口 |
| 连续验证失败 2-3 次 | 熔断，继续硬改可能扩大破坏面 |
| 你明确要求停下讨论 | 以你的新指令为准 |

除此之外，默认不问。**判断一个分叉要不要问你，唯一标准是“这两条路你最终看到的东西会不会不一样”**——一样的话，AI 自决；不一样，才列给你挑。

## AI 不应该问你的情况

下面这些属于内部实现，AI 自己定：

- 文件怎么命名。
- 函数放哪里。
- 用哪个 helper。
- 要不要抽小函数。
- 前后端字段怎么组织。
- 先改哪个文件。
- 选哪个测试命令。
- subagent 怎么派。

如果 AI 把这些拿来让你选，就是没有遵守“抓大放小”。

## 你怎么提需求更省事

普通说法就够：

```text
把 Dev Docs 和 agent 派工链路优化一下，让 AI 少问多做，并且过程留痕。
```

如果你想触发已有套路，可以在需求里多放关键词：

| 关键词 | 会触发的套路 |
|---|---|
| `route` / `api` / `接口` | 新后端接口规范 |
| `db` / `字段` / `schema` | 数据库加列规范 |
| `日志` / `埋点` | LogsView 起止日志规范 |
| `smoke` / `端到端` | smoke 验收脚本规范 |
| `badge` / `chip` | 前端标签/徽章规范 |
| `subagent` / `派工` | 团队 agent 派工规范 |

你不需要解释这些规范，Harness 会按任务名和关键词把相关 skill 注入给 AI。

## 做事痕迹在哪里看

| 你想看什么 | 去哪里看 |
|---|---|
| 这次任务大方向 | `dev/active/<任务名>/<任务名>-plan.md` |
| AI 判断会改哪些文件 | `dev/active/<任务名>/<任务名>-context.md` |
| 当前做到哪一步 | `dev/active/<任务名>/<任务名>-tasks.md` |
| UI 里的任务状态 | `dev/active/<任务名>/<任务名>-tasks.json` |
| 用户操作日志 | VibeSpace 的 LogsView |
| 后台任务状态 | VibeSpace 的 JobsView |
| 顺手发现的问题 | `dev/issues.md` |
| 自动沉淀经验 | `dev/memory/auto.md` |
| 你的长期偏好 | `dev/memory/manual.md` |

少说多做不是不留记录，而是把记录留在文件、日志和状态里，不靠聊天刷屏。

## Worktree 怎么理解

worktree 是 git 的临时副本。它的作用是让一个任务在独立工作区里改代码，避免多个 AI 会话互相踩脚。

建议策略：

| 任务类型 | 是否开 worktree |
|---|---|
| 改一个字、改一段文案 | 不开 |
| 单文件小改 | 可不开 |
| 多文件功能 | 默认开 |
| 多 agent 并发 | 必开 |
| 改数据库或核心流程 | 必开 |
| 不确定风险任务 | 必开 |

不是每个任务都要开 worktree。它是风险控制工具，不是仪式。

## 验收与归档

AI 完成后，交付摘要必须 ≤10 行，**第一行必须是验收指引**——告诉你“现在去哪里点哪里，看什么结果”。结构：

```md
验收：现在去哪里点哪里，看什么结果（一句白话，不带术语）。

改了什么：
验证过什么：
有没有遗留：
是否需要归档：
```

你只需照第一行去验收。看不懂第一行 = AI 没写合格，打回。验收通过后，在 Dev Docs 侧栏点归档。归档会触发后台 review job（review 任务，跑在后台不阻塞你），自动把可复用经验写入 `dev/memory/auto.md`。

## 工作流是不是真的生效

下面这几条是检查清单，平时跟 AI 配合时可以反向校验：

- 整个任务你**只在 plan 阶段点过一次头**，后面没被技术细节、命名选择、文件结构这种琐碎打扰。
- AI 给你的每段话都看得懂——没有“zod / mutation / payload / route handler”这种术语裸奔。
- 交付摘要第一行能直接拿去验收。
- 改动只跟你提的需求相关，没有顺手重构、顺手改无关代码。
- 如果某次 AI 来问你“函数放哪里 / 用 A 还是 B 库”，那就是没遵守——可以直接告诉它“你自己定”。
