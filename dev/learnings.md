# Learnings（跨任务可复用的经验）

> 单条尽量单行，格式：`- [<YYYY-MM-DD> / <来源任务>] <结论>（上下文：…）`。当某条经验被多次踩，再升级到 CLAUDE.md。

- [2026-04-29 / harness-worktree隔离] 隔离模式下 git worktree **不会**带 node_modules（被 .gitignore 忽略），所以隔离 session 不能直接跑 `pnpm dev` / `pnpm build` 类需要依赖的命令——这类工作流必须用 shared 模式。隔离 session 的预期用途是"agent 改源码"，build/test 留在主仓 session（上下文：worktree 的设计本意是"独立 working tree 共享 .git 仓库"，依赖目录是 working tree 的私有产物，本来就不该共享）
- [2026-04-29 / harness-worktree隔离] DialogHost 的三种 dialog（alert/confirm/prompt）不支持"主体 + 额外复选框"的复合形态。需要"先确认动作再追问一个布尔选项"时，分两次 confirmDialog 比扩展 DialogHost 加新 kind 更符合"外科式改动"原则——前者只动调用方，后者要修公共组件 + 改 ConfirmPending 类型 union（上下文：harness-worktree 隔离的关闭 session 流程需要"关闭? + 是否删 worktree?"两个问题，二段式 UX 影响很小）
- [2026-04-29 / harness-task绑定与jobs面板] ContextMenu 组件**已经原生支持 submenu**——`ContextMenuItem` 接口上就有 `submenu?: ContextMenuItem[]`，render 端有完整的 hover→子菜单展开逻辑。要做"行右键里一个二级动态列表"（如本任务"绑定到 session ▸ \<候选列表\>"）直接传 submenu 即可，不要再改 ContextMenu。最初 plan 时把它列为"扩展 ContextMenu 风险点"是没读源码就提防过度（上下文：本任务 context 阶段读源码后就消除掉了这条风险，提醒未来 plan 时对"风险点"先快速扫一眼源码再确定要不要列）
- [2026-04-29 / harness-task绑定与jobs面板] `addColumnIfMissing(db, table, column, def)` 这种"幂等加列"helper 已经在 db.ts 里——后续给 sessions 加新列直接复用：a) CREATE TABLE 里加；b) legacy CHECK 重建表也加（如果还在）；c) 末尾 addColumnIfMissing 一行兜底老 DB。**三处都要加**，不然新建库与升级库有差异（上下文：本任务给 sessions 加 task_name 时按这个三处套路加的，参照前一任务给 isolation/worktree_path/worktree_branch 加的格式）
- [2026-04-29 / harness-subagent卡片与按需技能] claude 的 `Task` 工具 subagent **不是 PTY 子进程**——它在 claude 主进程内跑，没有自己的 sessionId / stdin / stdout。所以"在 ProjectsColumn 里挂父子树形渲染"是死路；正确路线是通过 hook 抓 PreToolUse/PostToolUse 把 Task 调用记成"subagent run 卡片"挂在父 session 的 SessionView 上。下次有人提"父子 session"前先确认是不是同样的误区（上下文：本任务最初 plan 写的"父子 PTY 树"在 context 阶段读源码时被纠正成"虚拟 run 卡片"）
- [2026-04-29 / harness-subagent卡片与按需技能] hook payload 字段格式不要拍脑袋假设——实施时第一步**先 console.log 一次实际 payload 再 commit**。本项目里 `extractTaskInvocation` 用 `AIMON_HOOK_DEBUG=1` 守护一段 `console.log(JSON.stringify(payload, null, 2).slice(0, 4000))` 的写法是合理样板；后续要扩别的 hook 字段（比如 PostToolUse 的 tool_response 解析）按这个套路一次性看清字段名再写解析逻辑（上下文：A-2 步骤把"先 console.log 实际 payload 再 commit"写进 plan 当熔断点，避免猜错字段反复改）
- [2026-04-29 / harness-subagent卡片与按需技能] 简单 yaml frontmatter 30 行手写 parser 完全够用，**不要为它引 `gray-matter` 这种包**——只支持 `key: value` + 行内数组 `[a, b]` + 块数组 `\n  - a` 三种形态；嵌套对象 / 多行字符串场景再考虑换库。本项目 skills-service.ts 里的 parseFrontmatter 就是这个最小可用形态，下次再有"配置 md 文件"需求可参考（上下文：B-1 实施时我抗住了"为 30 行省事引一个新依赖"的诱惑）
- [2026-04-29 / harness-一键装配与团队面板] PermissionsDrawer 这种**叫 Drawer 但实际是居中 modal** 的命名跟实现不一致是项目历史；新加同类 UI 组件（HarnessTeamDrawer 等）**沿用同款居中 modal**，不要为新组件引入第二种布局形态（右侧滑出 vs 居中）。plan 里假设它是右侧滑出是错的——context 阶段读源码时纠正掉。下次设计前先 grep 一眼现有 Drawer / Dialog / Modal 的真实实现再决定形态（上下文：本任务 plan AS3 写"右侧滑出"，context 阶段读 PermissionsDrawer.tsx 发现 `fixed inset-0 z-40 flex items-center justify-center` 实际是居中 modal）
- [2026-04-29 / harness-一键装配与团队面板] 命令行脚本（install.sh / install.ps1）跟 server 端等价 TS 实现并存的"双实现"模式：跨语言不强行共享文件清单常量，**两边都加同步提醒注释**指向对方。这种模式的命门是"加新文件时记得改两处"，靠注释提醒不够稳——但**强行抽 JSON 清单 + 两边读** 又引入"再加一份语言无关的 manifest 文件"的复杂度。v1 取注释方案，等真发生过一次同步失误再升级到 manifest（上下文：本任务 install.sh / install.ps1 / harness-template-service.ts 三份独立实现，文件清单分别在三处定义；同步提醒注释加在三个文件顶部）
- [2026-06-03 / 飞书双向任务桥] 后端要"不经 HTTP 程序化起一个 session"（如桥/定时器/守护逻辑自动拉起总控台）时，**不要 fake 一个 fastify reply 去调 routes/sessions.ts 的 startSession**（它强耦合 reply 且夹带 worktree/skills 逻辑）。直接复刻它的 5 步核心即可：`createSession()` → `statusManager.onSpawn(id)` → `injectMcpForAgent(agent, projectPath, id, projectId)` → `ptyManager.spawn({sessionId,agent,cwd,env})` → `updateSessionPid(id,pid)`。`__hub__` 的 cwd/projectPath 都用 `getHubWorkspaceDir()`，injectMcpForAgent 会因 projectId===HUB_PROJECT_ID 自动走 injectHubMcps 注入 aimon-hub（上下文：feishu/hub-session.ts::ensureHubSession 就这么写的，避开了重构共享的 sessions.ts）
- [2026-06-03 / 飞书双向任务桥] 新增 aimon-hub MCP 工具的固定套路：a) 在 `mcp-hub/index.ts` 用 `server.registerTool(name, {title,description,inputSchema:zod}, handler)`，handler 里 `hubFetch('/api/hub/<x>', {method:'POST',body})`；b) 后端在 `routes/hub.ts` 加对应 `POST /api/hub/<x>`——挂这里而非 /api/feishu/* 是为了吃 index.ts 的 X-Hub-Token 鉴权 hook（仅守 /api/hub/*）。mcp-hub bin 是**独立进程走 HTTP**，不 import server 源码，所以工具里别直接引服务单例（上下文：send_feishu_message / send_input_to_session 两个工具都按此加，后者门禁=非hub+非shell+PTY活+waiting_input+1s无人类输入+短锁）
- [2026-06-03 / 飞书双向任务桥] Write/Edit 往源码里塞**字面控制字符的正则**（如 `/[\x00-\x1F]/` 想表达的范围）会被工具按字面字节落盘、反复 mismatch。要过滤控制字符就用**码点判断**（`for...of` + `ch.codePointAt(0) < 0x20`）或在正则里写 `\x` 转义而非真控制字节，源码里永远不留裸控制字节（上下文：feishu/inbound.ts 的 stripControls 改用码点循环，连吃两次字面字节 mismatch 后才换写法）
- [2026-06-05 / 工作流改独立文件引用] Claude Code 的 CLAUDE.md `@相对路径` 引用：每次会话自动展开进上下文、与内联同等强度、相对 CLAUDE.md 所在目录解析、递归最多 4 跳、项目内相对路径不弹授权框（仅 ~/ 或项目外才弹）。**坑**：块级 HTML 注释 `<!-- -->` 在注入前会被剥掉——所以版本戳之类要用"读磁盘文件"而非"读 AI 上下文"来检测。官方文档 code.claude.com/docs/en/claude-md.md#import-additional-files。

- [2026-06-12 / 启动脚本彻底清理旧进程] 本机 WMI 查进程必超时（Get-CimInstance Win32_Process 带/不带 -Filter、-OperationTimeoutSec 一律 30s+ 不返回），启动路径上的代码禁止依赖 Get-CimInstance/wmic 查进程。WMI-free 替代已验证：Get-Process（43ms，但拿不到父进程/命令行）、Toolhelp 快照 P/Invoke（毫秒级，拿 PID→PPID 树）、NtQueryInformationProcess（拿单个进程父 PID）、Process.StartTime（核验 PID 是否被复用）。现成实现见 scripts/start-cleanup.ps1。
