# 提交内容查看器 · Plan

## 大哥摘要（先看这段）

现在你在「源代码管理」面板底部那张**提交图**（一条条提交连成的时间线）里，点某次提交**什么都不会发生**——看不到这次提交到底改了哪些文件。这次就是把它补上：

- **点一次提交** → 能看到「这次提交改了哪几个文件」的清单（每个文件带颜色标记：改了 / 新增 / 删除，还有 +多少行 / -多少行）。
- **点清单里某个文件** → 在右边大编辑区打开这个文件**这次提交的具体改动**（绿色是新增的行、红色是删掉的行，也就是 diff＝两个版本的逐行对比）。

这是 VS Code、GitHub 这些工具里的标准做法，我照搬过来。**不动你任何现有数据，也不改提交本身**——纯粹是"看"的功能，随时点开随时关，不会误删误改任何东西。

**需要你拍板的只有一处**（在下面「需大哥确认的分叉」）：这份"改了哪些文件"的清单，是显示在**提交图里就地展开**，还是**在右边大编辑区单独开一页**。两种你看到的样子不一样，我给了推荐，你定。

## 需大哥确认的分叉（唯一确认点）

提交图在面板**底部、又窄又矮**。"点提交看文件清单"放哪里，有两种你能直接看到差别的做法：

- **方案 A（推荐）· 右边大编辑区开「提交详情」页**：点提交，右边主区域开一个标签页，顶部写这次提交的信息（作者 / 时间 / 说明），下面列改了哪些文件；点文件就在同一区域切到 diff。**好处**：地方大、文件再多也不挤，提交说明能完整看到，和你点普通文件看代码是同一块区域，习惯一致。这也是 GitLens / VS Code「查看提交」的主流形态。
- **方案 B · 提交图里就地弹出文件清单**：点提交，在那条提交旁边浮出一小块文件清单；点文件再去右边看 diff。**好处**：清单就在提交边上、更"贴着图"。**代价**：底部面板窄，文件一多就得在小框里滚动，提交说明也放不下；而且提交图的连线是按"每行固定高度"用绝对坐标画的，就地展开容易把下方连线顶歪，所以这块清单只能做成**浮层盖住**下方几行（不是把图撑开）。

我推荐 **A**：地方大、不挤、对非程序员更清楚，且完全不碰提交图那套画线逻辑（最稳）。如果你更想要"贴着图看"的感觉，回我一句切 B。

## 目标与验收标准

**目标**：让提交图里的每次提交可点开查看其改动文件清单，并能逐文件查看该次提交的 diff。

**可验收（在浏览器里照着点）**——以下按推荐的方案 A 描述（选 B 则文件清单出现在提交图浮层里，其余一致）：

1. 在「源代码管理」底部提交图，**点任意一次普通提交** → 右侧编辑区出现「提交详情」标签，列出该提交改动的文件，每行有状态标记（M/A/D/R）和 `+增/-删` 行数。
2. **点清单里某个文件** → 同区域显示该文件这次提交的 diff（新增行绿、删除行红）。
3. **重命名的文件** → 清单里显示 `旧路径 → 新路径`，点开 diff 不报错。
4. **删除的文件** → 点开默认进 diff 视图能看到整段被删内容（不会因为"当前已无此文件"而报错）。
5. **同一个文件分别在两次不同提交里点开** → 打开的是各自提交的 diff，不会串成同一份（标签不复用错版本）。
6. **快速连点提交 A 再点提交 B** → 最终显示的是 B 的文件清单，不会被 A 慢返回的请求覆盖（无竞态串台）。
7. **首次提交（没有父提交的根提交）** → 能看到清单/内容，不报 `from 缺失` 类错误。
8. **某次提交文件特别多 / 请求失败 / 仓库无提交** → 分别有"加载中""出错（可重试或有错误文案）""空"状态，不白屏、不把底部图撑乱。

## 非目标（本次不做）

- 不做提交搜索 / 过滤 / 按作者筛选。
- 不做"在提交详情里直接 checkout / revert / cherry-pick"等会改仓库状态的操作（本次纯只读查看）。
- 不重写提交图的连线 / 泳道布局算法（推荐方案 A 完全不碰它）。
- 不为二进制文件做专门的可视化 diff（沿用现有 FilePreview 行为，见边界）。

## 实施步骤（按推荐方案 A；选 B 则第 2、4 步换成 GitGraph 内浮层）

1. **store 加一处轻量缓存**：`commitDetailCache[projectId][sha] = CommitDetail`，按「项目+sha」分桶（避免切项目串数据）。SWR 风格，仿现有 `projectGraphCache`，不引新库。→ verify：类型检查过，缓存命中后重复点同一提交不再发请求。
2. **新增「提交详情」标签类型与视图**：给编辑器标签加一种 `commit` 类型（携带 projectId + sha），在 EditorArea 里识别并渲染新组件 `CommitDetailView`：调 `getProjectCommit(sha)` 取文件清单，复用 ChangesList 的状态徽标风格展示 `状态 + 路径 + +增/-删`，含加载/出错/空三态、列表区独立滚动、长路径中间截断+hover 全路径。→ verify：点提交后编辑区出现该标签并列出文件。
3. **文件行点击 → 开 diff**：点清单文件行调 `openFile({ projectId, path, ref:sha, from:parents[0], to:sha, commitSha:sha, status })`，复用既有 FilePreview/DiffView 渲染。根提交（无父）用 Git 空树 hash `4b825dc642cb6eb9a060e54bf8d69288fbee4904` 作 `from`。合并提交按第一父比较（与后端一致），视图里加一句很短的"与第一父提交比较"提示。→ verify：点文件出现 diff；重命名/删除/根提交/合并提交均按验收 3/4/7 表现。
4. **改写 GitGraph 的 onCommitClick**：从现在"把 selectedChange.path 塞成提交说明"的半空操作，改成"打开该提交的提交详情标签"。清掉因此不再需要的 `selectedChange.path=subject` 写法与那段 `void openFile` 占位注释。→ verify：点提交不再是无反应；`git diff --name-only HEAD` 只含白名单文件。
5. **竞态与边界收口**：CommitDetailView 内用 cancelled 标记防止 A/B 连点串台；`getProjectCommit` 失败有可见错误态。→ verify：连点两提交只显示后点的；断网点提交看到错误文案。
6. **类型检查**：`pnpm -F @aimon/web build` 成功（项目无单独 typecheck 脚本，build 即类型门禁，见 auto.md 经验）。

## 边界情况

- **根提交无父**：`parents[0]` 为空，用空树 hash 作 `from`，否则 diff 取不到基准。
- **合并提交多父**：统一与第一父比较（后端 commit detail 本就是第一父基准），UI 标注一句。
- **删除文件**：`ref=sha` 下文件已不存在，但默认进 diff 视图（from&&to 时 FilePreview 默认 diff tab），能看到删除内容；仅当用户手动切到"源码"tab 才可能空/报错，属可接受降级。
- **二进制文件**：DiffView 无专门二进制分支，空 patch 会触发 FilePreview 既有"无实际差异→自动回退源码/预览"逻辑（图片走 ImagePreview，其余可能空白）。本次不额外加二进制提示，沿用现状。
- **超大提交（数百文件）**：文件清单区固定最大高度 + 内部滚动，不撑乱底部图。
- **长路径 / 含空格中文#?的路径**：中间截断 + hover 显示全路径；请求走 URLSearchParams 统一编码（api.ts 现状已是）。

## 风险与注意

- **提交图 SVG 连线耦合**：连线按行 index × 固定行高用绝对坐标画。推荐方案 A 不碰它；若改选方案 B，文件清单必须做成不改变行高的浮层，否则下方连线会错位（Codex 重点提示）。
- **openFile 标签唯一键**：现有 key 由 `projectId+path+ref+from+to+kind` 组成，含 ref(=sha)/from(=父)/to(=sha)，天然按提交区分，不会串版本——实现时再核一次。
- **纯只读不埋日志**：展开/查看不属于会改系统状态的操作，按项目约定不加 logAction，避免日志噪音（manual.md 约定）。
- **假设**：`getProjectDiff(path,{from,to})` 对重命名(R)/复制(C)状态能取到正确 rename diff（path 用新路径）——这是"无需新增后端路由"假设里最易翻车处，实现时实测验收 3 覆盖。
