# 修复自定义CSS不生效 · Plan

## 大哥摘要

你在「外观」页粘贴那段米黄底深棕字的"古书风"CSS，日志说"apply-css 成功"，但页面没任何变化——这是 bug。

根因：我们仨预设主题（柔和深灰 / 亮色护眼 / 黑色玻璃拟态）在 CSS 里写成 `[data-theme="soft-dark"] { ... }`，比你粘贴的 `:root { ... }` 在 CSS 优先级（CSS specificity，浏览器决定哪条规则赢的内置打分）上更高，所以预设永远盖过你的自定义。

修完之后：你粘贴的 `:root { ... }` 会被自动加一点儿"权重"（变成 `:root[data-theme]`），就能反过来盖过预设主题，UI 会立刻变成你写的那个样子。改的是注入 CSS 时的一段小转换，不动你已存的主题数据，预设主题切换 / 重置 / 持久化（保存到浏览器本地）这些功能都不动。

## 目标

让用户粘贴的标准 `:root { ... }` 自定义 CSS 能盖过预设主题真正生效。

**验收标准（在浏览器里能看到）**：

1. 在「外观」页粘贴本次问题里的"古书风"CSS（含 `:root { --color-bg: 244 236 220; --font-sans: Georgia,...; --radius: 0; ... }`），点应用后**整个 UI 立刻**变成：米黄底（不是深灰）、深棕字、衬线字体（Georgia）、所有圆角变方角。
2. 切预设主题（柔和深灰 / 亮色护眼 / 黑色玻璃拟态），自定义 CSS **仍然生效**——因为它优先级被提升过，会持续盖过预设。
3. 在 LogsView 仍能看到 `scope=theme action=apply-css` 的起止配对（开始 + 成功），不退化。
4. 在「外观」页点"重置自定义 CSS"，UI 立刻回到当前预设主题（米黄被清掉）。
5. 刷新页面，自定义 CSS 仍然在 → 不闪一下预设再切回来（因为 `index.html` 的同步兜底脚本也走同样的转换）。

## 非目标

- 不重写主题系统，不改预设主题的命名 / 数据结构 / 切换流程。
- 不在前端解析整段 CSS 做语法/安全校验（仍走现有 100KB 上限保护）。
- 不处理 `html { ... }` 这种等价但少见的根选择器（只处理 `:root`，CSS 主题的业界惯例就是 `:root`）。
- 不改「外观」页 UI、不动 LogsView 渲染。

## 实施步骤

1. 在 `packages/web/src/theme/store.ts` 加一个纯函数 `bumpRootSpecificity(css: string): string`：
   - 把裸 `:root`（后面紧跟 `{` / 空格 / `,` / `:` 等选择器分隔符）替换为 `:root[data-theme]`，让特异性从 (0,0,1) 升到 (0,1,1)，盖过预设主题的 (0,1,0)。
   - 已经写成 `:root[...]` 或 `:root(...)` 的不动（用负向预查跳过）。
   - **验证**：在 dev console 用几段样本 CSS 跑函数，输出符合预期。

2. 让 `applyUserCss(css)` 在写入 `<style>` 之前先跑 `bumpRootSpecificity`。
   - **验证**：浏览器粘贴问题里那段 CSS，检查 `<style id="user-theme">` 的 textContent 已经是 `:root[data-theme] { ... }`，UI 立刻变样。

3. 在 `packages/web/index.html` 的同步 IIFE FOUC 兜底脚本里复刻同一个正则转换（脚本不能 import store.ts，得就地一行）。
   - **验证**：刷新页面，UI 不闪一下预设、直接呈现自定义 CSS 效果。

4. 跑项目层面的 TypeScript 类型检查。
   - **验证**：`pnpm -C packages/web typecheck`（或仓库根的同等命令）成功。

## 边界情况

- CSS 注释 `/* :root */` 内的命中 → 仍在注释里，无害。
- CSS 字符串字面量含 `:root` → CSS 里没有这种合法语义，忽略。
- 用户故意写 `:root[data-theme="custom"]` → 负向预查跳过，原样保留。
- 空字符串 / 无 `:root` 规则 → 正则不匹配，原样返回（包括重置时 `applyUserCss('')`）。
- 多行 / 多块 `:root` 规则 → 全局替换都升格。
- `:root:hover { ... }` 这类已经带伪类的 → 特异性已经够（0,1,1），不需要再升，不动也不影响行为；但当前正则会把它替换成 `:root[data-theme]:hover`——结果仍然语义正确，只是冗余一个属性选择器，无副作用。

## 风险与注意

- **假设**：用户粘贴 `:root` 的语义就是"覆盖根 token"。如果有人反着用（粘贴 `:root` 但不想盖过预设），会受影响——目前没人这么用，可接受。
- 不影响 LogsView 操作日志（已埋）、localStorage 持久化、预设主题切换、shikiVersion 触发的代码块重渲染。
- inline 脚本和 store.ts 各持一份等价正则（不能共享代码因为 inline 是同步 IIFE）→ 后续若改转换规则要两处同步改。在两处都加一行注释互相指向。
