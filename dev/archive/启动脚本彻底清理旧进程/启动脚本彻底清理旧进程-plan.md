# 启动脚本彻底清理旧进程 · Plan

## 大哥摘要

每次双击 start.bat，先把上一次启动留下的所有"自己人"清干净——包括上次没关的黑色命令行窗口、还在后台跑的服务进程（node：跑 JavaScript 的程序）、以及编译小工具（esbuild：前端打包用的辅助程序）——然后再启动全新的一套。做完后你双击 start.bat 永远是干净的新实例，不会出现"旧窗口堆一桌""端口被占启动失败"的情况。不动任何数据和界面。

> memory 扫过：auto.md / manual.md 无与启动脚本进程清理直接相关的条目。

## 目标

双击 start.bat 时，上一轮启动遗留的全部相关进程被清掉，本次启动是全新进程。

验收标准（大哥可操作）：
1. 双击 start.bat 启动 → 不关窗口，再双击一次 start.bat → 第一个黑窗口自动消失，新窗口正常启动到服务可用（浏览器能打开面板）。
2. 第二次启动的输出里能看到 `killing stale ... PID ...` 字样，证明清理动作真的发生了。
3. 端口不再报"被占用"。

## 非目标

- 不改 pnpm dev 脚本本身、不动 server/web 业务代码。
- 不做开机自启 / 服务化（Windows 服务）。
- 不清理与本项目无关的 node 进程（别的项目、编辑器插件照常活着）。

## 实施步骤

1. 新建 `scripts/start-cleanup.ps1`：一次性查全系统进程快照，按三条规则圈出要杀的：
   - 旧的 cmd.exe 且命令行里含本 start.bat 完整路径（= 上次双击的窗口）；
   - node.exe / esbuild.exe 且命令行里含项目根路径（大小写不敏感）；
   - 以上进程的所有子孙进程（这样 pnpm 主进程、相对路径启动的子进程一网打尽）。
   同时保护"当前这条链"（本次的新窗口、新 powershell、explorer 祖先）不被误杀。
   → 验证：脚本语法能跑通，手工执行不误杀当前窗口。
2. 改 start.bat：把清理挪到 `pnpm install` **之前**（旧进程锁着 node_modules 文件时 install 会失败），调用新 ps1 替换原来的单行清理；端口清理保留作兜底。
   → 验证：连续双击两次 bat，旧窗口关闭、新实例正常起服务。

## 边界情况

- 用户在已开的终端里手敲 `start.bat`（不是双击）：该 cmd 命令行里可能没有 bat 完整路径，窗口不会被关，但里面的服务进程仍会被"项目路径匹配 + 子孙展开"清掉，端口兜底也在。可接受。
- 命令行里带项目路径的**其他工具**（如某些编辑器的索引进程是 node 且参数里写了本项目路径）会被一并杀掉——这是现有脚本就有的行为，本次未扩大到 node/esbuild 之外。
- WMI 卡死：保留 30 秒超时，查不到就跳过清理继续启动。
- stable 副本（路径含 stable）与 dev 副本路径不同，互不误杀。

## 风险与注意

- 杀进程按"先圈集合再统一杀"，极小概率 PID 复用误杀，窗口期毫秒级，可接受。
- bat 文件必须保持 ASCII（cmd 解析非 ASCII 会乱），注释用英文。
