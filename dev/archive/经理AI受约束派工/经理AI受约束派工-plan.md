# 经理AI受约束派工 · plan

## 大哥摘要（先看这段）

给 VibeSpace 加一个**「项目经理 AI」**:你给它一个大目标,它把活拆成几张任务卡,分给几个互不干扰的终端 AI(各自在 git 隔离副本里干,互不踩脚)并行做,关键处停下来等你一键放行。配套加一个**「经理 AI 边界」设置页签**——普通开关(同时干几个、要不要先给你看任务图)随手调;危险开关(让 AI 动数据库、删文件、花钱)默认锁死,点开时弹白话风险提示,你确认了才生效。

**关键安全保证**:这套东西的"危险动作拦截"**不靠经理 AI 自觉**——是**后端自己检查 AI 实际干了什么**(比如真删了文件、真碰了数据库),AI 标错了也拦得住。这样你这个不读代码的人,不用靠"看起来像描述"去判断安不安全,机器替你把关。

你能在哪验收:**设置弹窗里多一个「经理 AI 边界」页签**;给经理 AI 一个目标后,它会**先把整张任务图摆给你看、停下等你点"开始"**才派工;干完给你白话总结。

要不要担心动到现有东西:**不会**。新加一层,现有手动派活/子任务/看板全照旧;危险动作默认全锁死;"自动合并"第一版不开放,永远人工放行才合并。

## 目标

让一个常驻"经理 AI"接管现在**人手动做的两步**——写任务拆分图、点"派子任务"——并在干完后汇总;同时给出**后端硬约束**的边界(危险动作默认锁死、靠实际改动检测而非 AI 自报)。

### 验收标准（可验证 / 含浏览器可观察项 / 含"防假绿灯"项）

1. **边界设置页签**:打开设置弹窗,出现「经理 AI 边界」页签;普通区(并发 1–3、派工前确认任务图、出错即停)+ 危险区(允许动数据库/删文件/付费,默认全关;"自动合并"灰显标注"第二版开放")。保存后刷新仍在(落盘 `app-settings.json`)。
2. **默认全保守**:全新环境(没有 app-settings.json 或缺字段)读出来的危险项**全是关**、并发=2、确认任务图=开——不只是"默认值写在代码里",而是 GET 接口真返回保守值。
3. **危险项确认门**:危险区点开关想打开时,弹白话风险确认("打开后 AI 可改你的数据/删文件且不再问你,可能无法撤销,确定?");**取消则开关不变、不落盘,刷新仍为关**;确认才真打开。
4. **派工前任务图确认(后端强制,非仅提示)**:经理 AI 拆出图后,后端发一个绑定该图内容的"确认凭证";UI 显示"经理 AI 拆出 N 个任务,确认后开始";未带有效凭证调 dispatch → 后端**拒绝**。**改了图 → 旧凭证作废**(确认 A 图无法派 B 图)。`managerConfirmGraph` 关时跳过此门。
5. **并发上限生效(项目级)**:并发设 2,同一项目即使同时有多张图,后端实际同时只跑 2 个 worktree(dispatch 处 `countActive(projId)` 比对 `managerConcurrency`)。
6. **危险动作靠"实际改动"硬拦(核心防线)**:
   - 删文件:子任务跑完用 `git diff --name-status` 检测真实删除;若有删除且"允许删文件"为关 → **该子任务不得进入 review-ready(直接标失败/阻断),不被合并**。
   - 动数据库:按 DB 文件(`*.db`/`data/`)、迁移路径、diff 里 SQL/schema 关键词检测;命中且边界关 → 阻断。
   - **假绿灯防御**:构造一个**没标 danger 但实际删了文件 / 改了 DB**的子任务,验证它照样被后端拦下、进不了合并。
7. **裸 HTTP 绕过测试**:直接 curl 打 dispatch(不经过经理 AI)——未带确认凭证 / 触碰未授权危险动作的任务,后端**照样拒绝**。证明约束在后端不在提示词。
8. **跳过即阻塞下游**:某子任务因危险被跳过 → 依赖它的后续子任务**不被 `wireWaveAdvancer` 自动派发**,标阻塞并在 UI/日志可见。
9. **出错即停作用域**:`出错即停`开 → 某子任务失败时,该图**所有未派发的波次停止**(不只停失败那个);关 → 独立分支继续。
10. **完工汇总**:全部子任务合并后,经理 AI 写白话总结(改了什么、去哪验收)。
11. **操作日志**:派工/确认任务图/保存边界/危险项开启,LogsView 有 `scope=manager` 起止配对;拒派/阻断危险动作有 WARN/ERROR 条目(人工触发验证过)。

## 非目标（本轮不做）

- **不做全自动撒手**:不绕过确认门、不在边界关闭时强行干危险活、不自动合并。
- **不做自动合并**:第一版始终"人工 approve 才合并"(现有行为不变);自动合并留第二版。
- **不做自主反思循环(soul flow)**:经理 AI 只在(a)你给目标、(b)子任务状态变化两个时机被驱动,不"空闲自己找活"。
- **不做付费动作的硬检测**:付费最难机检,第一版**默认禁止 + 不给 worker 配付费入口**即兜住,不靠关键词猜(toggle 保留但标"弱约束")。
- **不做 agent 间双向邮箱(s09)**、**不做跨项目编排**、**不改现有手动派活/子任务/看板行为**。

## 实施步骤

> 顺序原则(采纳 Codex):**先把后端硬约束做出来并能独立验收,再接经理 AI**——否则会出现"看着受约束、实则全靠提示词自觉"的空壳。Phase 1/2 是后端真护栏,Phase 3 才是经理 AI 消费它。

### Phase 1 · 边界设置（UI + 持久化,可独立验收）

1. **后端加边界字段**(四处同步:`app-settings.ts` 接口 + 默认 + clamp + `routes/app-settings.ts` 的 zod `UpdateBody`):
   - 普通:`managerConcurrency`(1–3,默认 2)、`managerConfirmGraph`(默认 true)、`managerStopOnFailure`(默认 true)
   - 危险:`managerAllowDbChanges`、`managerAllowFileDelete`、`managerAllowPaidCalls`(默认 false)
   - verify:`pnpm -F @aimon/server build` 过;删掉 app-settings.json 后 GET 返回全保守值(验收 2)。
2. **前端类型 + store + 拉取**:`web/src/types.ts` 镜像、`store.ts` state/默认/setter、`App.tsx` 启动拉取。
3. **设置弹窗加页签**:`SettingsDialog.tsx` 的 `SettingsTab` + `SETTINGS_TABS` 加 `'manager'`;右侧条件渲染普通区(照「AI 终端数量上限」控件)+ 危险区(每个开关点开先弹确认框,"自动合并"灰显);保存用 `logAction('manager','save-boundary',...)`。沿用 `h-[600px]` 固定高、内部滚动(manual.md 偏好)。
   - verify:`pnpm -F @aimon/web build` 过;浏览器看页签/改值保存/刷新仍在/危险项确认框取消不落盘(验收 1/3)。

### Phase 2 · 后端硬约束（核心防线,可用裸 curl 独立验收）

4. **确认凭证机制**:dispatch 路由前置一个"提交任务图→拿凭证"的步骤(可在现有 dispatch 路由加 `confirmToken` 参数 + 一个轻量内存表:`projectId+task → {graphHash, token}`)。`managerConfirmGraph` 开时,dispatch 必须带与当前图 hash 匹配的有效 token,否则拒绝;图变→hash 变→旧 token 失效。
5. **并发上限 enforce**:dispatch 入口(`task-subtasks.ts:369–404`)读 `getAppSettings()`,用 `managerConcurrency` 替换 `DEFAULT_CONCURRENCY`,基于 `countActive(projId)`(项目级,跨多图)。每推进新波次前**重新读设置**(中途改并发能生效)。
6. **危险动作硬检测(不信 danger 自报)**:
   - `SubtaskSpec` 加可选 `danger?` 字段**仅作 UI 提示/排序**,不作授权依据。
   - 子任务 verify 阶段后、进 review-ready 前,加一道**实际改动检测**:`git diff --name-status`(基于该 worktree 分支)→ 有删除且 `managerAllowFileDelete` 关 → 阻断;diff 命中 DB 文件/迁移/SQL 且 `managerAllowDbChanges` 关 → 阻断。**fail-closed**:检测异常/无法判定 → 当作命中,阻断。
   - 阻断 = 标失败 + `serverLog('error','manager',...)` + 不进 review-ready、不可合并。
7. **跳过即阻塞下游**:某子任务被阻断/跳过 → `wireWaveAdvancer` 不得派发其 `depends_on` 后代;标阻塞。
8. **出错即停作用域**:`managerStopOnFailure` 开 → 任一子任务失败/被阻断时,停该图所有未派发波次。
   - verify(全部用裸 curl,不经经理 AI):带错/不带 token 被拒(验收 4/7);构造未标 danger 但真删文件/改 DB 的子任务被拦(验收 6);被跳过任务的下游不自动派(验收 8);并发多图仍≤2(验收 5);出错即停范围对(验收 9)。`pnpm -F @aimon/server build` 过。破坏性变更协议:改 `SubtaskSpec` 前后 grep 引用确认无残留。

### Phase 3 · 经理 AI 角色（消费已硬化的后端）

9. **写经理 AI skill** `.aimon/skills/经理AI受约束派工.md`(triggers:"经理"/"派工"/"拆活"):SOP =
   - 收目标 → 调研 → 写 `plan.md` 含自拆图(每个子任务标 `write_files`、可选 `danger` 提示、一句"做完用户在哪看到效果")
   - 提交图拿确认凭证 → 把任务图用白话摆给大哥、停下等确认(`managerConfirmGraph` 开时)
   - 确认后 Bash `curl` 打 `dispatch-subtasks`(带 token + maxConcurrency)
   - 轮询子任务状态(走现有状态机,**不扫 PTY buffer**——auto.md 2026-06-02 教训),全 review-ready 后停;**绝不自动合并**(等大哥点);全合并后写白话总结
10. **入口(最省形态)**:复用现有"起 claude 会话 + 绑 task + 注入 skill",第一版不造新 UI 按钮;任务名含触发词即注入经理 skill。若验收觉得不顺再单独加"召唤经理"按钮(后续)。
    - verify:给经理 AI 一个真实小目标(2 个互不依赖、可回滚、无危险的活)→ 出图→停下确认→派工→并发≤2→停 review-ready→大哥合并→它总结。
11. **同步 blueprint**:`dev/agent-team-blueprint.md` "不在 blueprint 范围"里"自动派工 s11 已劝退"一句,改为指向本任务(避免规则漂移,manual.md 教训)。

## 边界情况

- **danger 解析失败/未知类型/缺字段** → fail-closed,当危险处理。
- **自拆图非法/有环** → 现有 `validateGraph` 拦,经理 AI 回去改图。
- **并发设 1** → 串行,波次每次派 1。
- **设置字段没同步 zod** → 被静默剥掉(代码注释已警告),四处必同步,Phase 1 verify 往返测。
- **图改后旧凭证复用** → hash 不匹配,拒绝。
- **worker 在隔离副本里删文件但最终没被合并** → 不影响真项目(隔离兜底);但仍按边界在合并前拦(双保险)。

## 风险与注意

- **决定性风险(Claude+Codex 双评一致)**:能不能用,在**任务能否被准确拆开 + verify/硬检测能否识破"假成功"**,不在那个同意按钮。第一版只跑"边界清楚、验收明确、可回滚"的活。
- **非程序员特有风险**:你凭"看起来像描述"验收 → 闸口提示必须写"你会看到什么变化/最坏损失什么",不能只写技术动作名(验收 3/6 已要求)。后端硬检测是对此的核心补偿。
- **付费弱约束**:第一版付费靠"默认禁止+不配付费入口",不做内容检测,handoff 要如实说明这条是弱约束。
- **破坏性变更**:Phase 2 改 `SubtaskSpec`(跨文件导出类型)+ dispatch 关键链路,按破坏性协议 grep 前后核对。
- **量化(第二版上面板,本版人工记)**:省了多少派工、一次验收通过率、合并后返工率。不能稳定省时间就是"批量造待办的按钮"。
