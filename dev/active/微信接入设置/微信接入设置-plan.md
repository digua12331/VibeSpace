## 大哥摘要

这次会在 VibeSpace 的「设置 → 微信机器人」里加上取码、扫码连接、绑定本人和连接状态，手机扫码后就能看到是否连上。
连上后，您可以从微信给总控台 AI（帮您查看项目、派任务和回答问题的主会话）发纯文字，AI 的回答会回到同一个微信对话。
首版只允许设置页里绑定的本人使用，只做即时一问一答；主动提醒仍走飞书，微信页会明确提示这个限制。
不会改现有项目数据和飞书行为；现有界面只会多一个「微信机器人」页签，退出微信连接会清掉微信登录凭证和收消息进度，但保留已绑定的本人账号。

## 目标

解决微信取码、扫码登录、连接状态和总控台双向问答目前分散且不可直接使用的问题，并以一个人使用的本地工具为前提，完成最小够用的安全闭环。

可验证的验收标准：

1. 在浏览器打开「设置 → 微信机器人」，点击「取码连接」后能在页签内看到二维码；手机扫码确认后，状态从「等待扫码」变为「已连接」；二维码过期或凭证失效时显示「连接异常」和重新取码提示。
2. 扫码成功后，设置页提供短时 owner 绑定窗口（owner，即绑定的本人账号）和一次性绑定口令；只有在窗口内发送正确口令的微信号会被绑定，绑定前的消息不进入总控台。
3. 绑定本人从微信发送纯文字后，总控台终端能看到带 `[微信 requestId=...]` 标识的内容；总控台 AI 通过 MCP 工具（让总控台 AI 调用后端动作的能力）按该 `requestId` 回复，微信能收到对应回答。
4. 首版严格串行：上一条微信问题尚未回复时，再发一条会收到“上一条仍处理中，请稍后”的提示，不会把回答发错对话；非 owner 微信号不能把消息转入总控台，也不能收到总控台回复。
5. 浏览器 LogsView（操作日志页面）能看到 `scope=wechat` 的 `login`、`bind`、`inbound`、`reply`、`logout` 起止配对；人工触发二维码过期、非法绑定口令或失效凭证后，能看到对应 ERROR 日志。
6. 自动测试覆盖重复取码、单一收消息循环、停止中的请求、`-14` 凭证过期、收消息进度提交与消息去重、总控台会话并发创建和重启恢复；`pnpm -F @aimon/server build` 与 `pnpm -F @aimon/web build` 均通过。

## 非目标

- 不做微信主动推送、任务完成提醒、群聊、图片、语音或文件；主动提醒继续走飞书。
- 不抽象“多通道框架”，不改变现有飞书桥的用户行为；微信发送入口和 MCP 工具按现有飞书模式显式新增。
- 不做微信凭证加密、自动探活或无法可靠判断的“消息已送达”展示；只显示连接状态和发送请求结果。

## 实施步骤

1. 把 `ensureHubSession` 从飞书目录整理为通道无关的总控台会话模块，飞书与微信共同复用；加入单飞保护（同一动作同时只允许一个在跑），确保并发消息只创建一个总控台会话，并保留飞书现有的重启通知行为。  
   verify: grep（全项目搜索引用）确认飞书和微信都只调用共享的 `ensureHubSession`；并发测试证明只创建一个会话；飞书现有连接与通知测试不回归。

2. 新增单一 `WechatClient`（微信连接管理器），统一负责取码、扫码状态轮询、登录恢复、收消息长轮询（一次请求最长等待约 35 秒收消息）、回复和停止，不再拆出第二套登录状态机。连接状态只保留 `Idle / Scanning / LoggedIn / Error` 四态。  
   配置文件只保存开关、登录凭证、经过校验的微信 iLink 地址、游标（收消息进度）和 owner 绑定；二维码、连接状态、活跃登录代次、当前回复上下文和最近收发时间只放内存或临时目录，不写入 `wechat.json/config.json`。  
   verify: 模拟请求测试证明重复点击取码只保留一个活跃登录任务；登录与收消息循环均为单飞；`stop/logout/disable` 会中止正在等待的请求并忽略旧代次结果；运行态不会写入配置文件。

3. 完成微信连接生命周期的最小保护：取码时用代次编号和请求中止器取消旧任务；二维码过期后清除活跃任务并显示重新取码；`getupdates` 与 `sendmessage` 遇到 `errcode=-14` 时统一停止连接并进入 `Error`；普通网络异常使用有上限的退避重试，不让单次异常退出整个循环。服务启动时仅在开关开启且凭证完整时从已保存游标恢复，服务关闭前显式 `disconnect`。  
   verify: 自动测试覆盖二维码过期、重复取码、网络异常、两条路径的 `-14`、重启恢复和关机中止；人工重启服务后确认已登录会自动尝试恢复，失效凭证会提示重新扫码。

4. 建立仅 owner（绑定的本人微信号）可用的入站与回复闭环：扫码后由设置页开启短时绑定窗口并生成一次性口令，绑定前任何消息都不转发；绑定成功后只有 owner 的纯文字消息会进总控台，AI 的回复也只发回 owner 这个对话。每条被接收的消息生成稳定 `requestId`（本次问答的编号），按消息 ID 去重，先完成处理再提交游标；写入总控台时带 `[微信 requestId=...]` 前缀。  
   首版同一时间只允许一个待回复请求，回复上下文按 `requestId` 存在内存；新增 `send_wechat_reply(requestId, text)` MCP 工具和对应 Hub 回复接口，确保 AI 只能回复当前有效请求。非文本消息回复“不支持该类型”并正常推进游标。  
   verify: owner、非 owner、错误绑定口令、连续两条消息、重复消息、非文本消息和崩溃重启后的重复投递均有测试；浏览器总控台能看到通道标识，微信实际收到与 `requestId` 对应的回答。

5. 新增机器级微信接口和启动入口：提供状态读取、取码连接、开始绑定、退出登录和显式重置绑定等 `/api/wechat/*` 接口；普通退出保留 owner，只有“重置绑定”会清除 owner。在服务器启动时尝试恢复微信桥，在服务器关闭时先停止微信桥。登录内容以 base64（把内容转成可传输文本的编码）返回给前端，由前端生成并显示二维码，不在后端生成或保存图片文件。所有会改变连接状态的接口用 `serverLog` 记录起止配对。  
   verify: 接口校验拒绝非法参数与非 HTTPS/非微信 iLink 地址；成功与失败分支均有 `scope=wechat` 起止日志；后端目录中不产生二维码图片文件。

6. 在 `SettingsDialog` 增加固定高度、内容区内部滚动的「微信机器人」页签，并在前端接口封装中接入状态、取码、绑定、退出和重置绑定操作。页面显示四态状态、二维码、绑定倒计时、已绑定本人、最近收发时间，以及“仅即时问答、主动提醒仍走飞书、收不到回复时重新取码”的说明。取码、绑定、退出和重置绑定操作使用 `logAction` 记录起止配对。  
   verify: 浏览器逐项点击验收取码、扫码、绑定、退出、重置绑定和二维码过期；普通退出后 owner 仍保留，重置绑定后原 owner 不能继续使用；切换页签时设置弹窗高度不变化；LogsView 能看到前端操作日志，并人工触发至少一个失败分支的 ERROR。

7. 完成端到端验收和回归：用真实微信完成“取码 → 扫码 → 绑定本人 → 发消息 → 总控台处理 → 微信收到回复”，同时验证非 owner 拒绝、上一条处理中时拒绝第二条、失效凭证重新登录、服务重启恢复和飞书行为不变。  
   verify: 保存浏览器可观察结果与关键日志；运行 `pnpm -F @aimon/server build`、`pnpm -F @aimon/web build` 及本任务新增测试，全部通过。

## 边界情况

- 同时重复点击“取码连接”时，旧二维码和旧扫码轮询必须立即失效，只有最新二维码能完成登录。
- 二维码约 3 分钟过期；过期后进入 `Error` 并提示重新取码，不继续后台轮询。
- 同一时间只处理 owner 的一条待回复消息；第二条消息会被明确拒绝，避免回复上下文串线。
- 非 owner、绑定窗口外发送绑定口令、错误口令和绑定前普通消息都不能进入总控台。
- 非文本消息不进入总控台，但会收到“不支持该类型”的回复，并正常推进收消息进度。
- 收消息处理完成后再提交游标，并按稳定消息 ID 去重；这样崩溃重启最多造成可识别的重复，不会静默丢消息。
- 服务重启后可以从持久化凭证和游标恢复收消息，但内存中的待回复上下文不会恢复；重启期间未完成的问题需由 owner 重新发送。
- `getupdates` 或 `sendmessage` 返回 `-14` 时统一进入需重新扫码状态；网络抖动、超时、非法返回和限流使用有限退避，不制造请求风暴。

## 风险与注意

- 主动推送不可能：微信官方 ilink 协议是拉取式设计，机器人只能借用户来话附带的 context_token（回信凭据）回话，没有来话就发不出消息——试点 9 次实测全部无声失败，5 个独立开源拆解一致确认。所以"任务完成主动提醒你"这类通知必须继续走飞书，微信只能做"你问 AI 答"。
- 出站无声失败：发送接口失败时照样返回"成功"（HTTP 200 空响应），没有任何报错信号，程序自己无法得知消息已丢失（试点实测）。界面与日志只能如实显示"发送请求已受理"，不能承诺"已送达"。
- 出站会话寿命：**已解决** ✅。试点第四轮验证：sendmessage 带上 `client_id`（每条消息一个唯一去重编号）后，登录 25 分钟以上、静置 15 分钟以上，回复仍正常送达；前三轮不带 client_id 时出站通道 10-15 分钟内必死。实现中所有发消息调用必须带唯一 client_id。token 更长期（小时/天级）的寿命仍未知，实施中靠"-14 统一处理 + 界面提示重新取码"兜底。
- 对“发送成功”的最小检查只验证返回结构、`errcode` 和请求是否被微信接口接受；即使接口返回成功，也不能向用户承诺消息已实际送达，页面和日志必须使用“发送请求成功”措辞。
- 腾讯条款 4.7 保留随时限速、拦截或封禁第三方 AI 接入的权利；该通道应保持可关闭，失败不能影响 VibeSpace 主功能或飞书桥。
- owner 绑定是安全边界：只有在设置页主动开启的短时窗口内完成口令确认的微信号可以绑定；重新绑定必须显式执行“重置绑定”并重新取码，不能由任意第一条消息自动抢占。
- `baseUrl` 保存和使用前必须限制为 HTTPS 且属于允许的微信 iLink 域名，避免凭证被发往非微信地址。
- 共享总控台会话模块会触及飞书现有引用；实施前需列出引用影响，修改后再次全项目搜索旧引用，并以飞书回归结果作为完成条件。
- 运行态不持久化意味着进程重启会丢失二维码、状态和待回复上下文，这是首版为避免配置污染与回错消息接受的取舍。
- 克制原则：这是给一个人用的本地工具，按“单一客户端、单一 owner、单一待回复请求”实现；不为将来扩展增加通道框架、配置点或投机性错误处理。
