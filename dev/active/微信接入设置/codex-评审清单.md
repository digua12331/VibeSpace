# 微信接入设置 · Codex 评审清单（会审第一道，原文存档）

[简化] `login.ts` 与 `client.ts` 都管理轮询生命周期，容易形成两套状态机 → 合并为单一 `WechatClient`，路由只调用取码、启动、停止方法。
[简化] `connected` 与 `polling` 用户含义重叠 → 状态精简为 `off | waiting_scan | polling | failed`，不要持久化运行状态。
[简化] `lastContextToken`、`lastInboundAt` 属于临时运行数据，却被放进配置文件 → 配置只保存凭证、开关、白名单和游标；临时回复上下文留内存。
[简化] 微信直接导入 `feishu/hub-session.ts` 会造成模块归属和日志混乱 → 将总控台会话逻辑移成通道无关模块，飞书和微信共同复用。
[简化] 二维码渲染保留前后端两条候选路径会扩大范围 → 固定由前端渲染登录链接，不新增后端图片接口。
[简化] 暂时只有两个消息通道，不值得抽象完整"多通道框架" → MCP 工具和发送端点按现有飞书模式显式新增即可。
[风险] 全局保存"最近一条消息"的用户与 `context_token`，连续消息或多人消息会让 AI 回错人 → 回复必须绑定具体入站请求，或首版严格串行且仅允许 owner。
[风险] 扫码后第一条入站消息自动成为 owner，任何抢先发消息的人都可能完成绑定 → 增加短时绑定窗口和明确确认步骤，绑定前不转发到总控台。
[风险] 重复点击取码可能启动多个扫码状态轮询，旧二维码确认后可能覆盖新凭证 → 只允许一个活跃登录任务，并用代次编号和 `AbortController` 取消旧任务。
[风险] 登录确认、配置保存和服务启动都可能调用 `start()`，从而产生多个长轮询循环 → `start/stop` 增加单飞锁或代次保护，确保任意时刻只有一个循环。
[风险] 单纯设置停止标志无法立即结束最长 35 秒的请求，旧请求返回后还可能写游标 → `stop/logout/disable` 必须主动中止正在进行的 `fetch` 并忽略旧代次结果。
[风险] 草案只说明 `getupdates` 遇到 `-14` 断开，但 `sendmessage` 同样可能返回 `-14` → 两条路径共用"凭证过期"处理，停止轮询并将状态置为需重新扫码。
[风险] 协议会出现 HTTP 200 空对象但微信实际未收到的无声失败 → UI 和日志只能表述"发送请求成功"，不能显示"消息已送达"。
[风险] 游标在处理消息前落盘会丢消息，处理后落盘则崩溃重启可能重复投递 → 明确提交顺序，并像飞书入站一样按稳定消息 ID 去重。
[风险] `ensureHubSession()` 当前没有并发保护，飞书与微信同时触发时可能各自拉起一个总控台 → 在共享总控台会话模块加入单飞创建保护。
[风险] 飞书和微信都把纯文本直接写入同一个总控台终端，消息可能交错且 AI 不知道回复来源 → 入站文本附带通道和请求标识，并据此选择回复上下文。
[风险] `hub-session.ts` 的重启通知钩子只有一个，若微信也注册会覆盖飞书通知 → 保留飞书通知所有权，或改为多个订阅者。
[风险] 服务关闭流程当前未停止桥接客户端 → 微信长轮询应在 `app.close()` 前显式停止，避免关机期间继续写配置和打错误日志。
[遗漏] 文件清单未包含微信桥启动入口、服务器启动注册、Hub 回复路由和 MCP 工具文件 → 在实施范围中补齐 `wechat/index.ts`、`server/index.ts`、`routes/hub.ts`、`mcp-hub/index.ts`。
[遗漏] 服务重启后的自动恢复路径未明确 → 启动后仅在开关开启且凭证完整时恢复长轮询，并从持久化游标继续。
[遗漏] owner 自动绑定后与 `allowUserIds` 的关系不明确，可能绑定成功却仍被白名单拒绝 → 明确 owner 永远允许，或绑定时自动加入白名单。
[遗漏] `logout` 对凭证、游标、owner 和白名单的数据影响未定义 → 登出清除 token、baseUrl、游标和临时上下文，默认保留 owner 与白名单。
[遗漏] 二维码过期后的服务端和界面行为未写清 → 到达 `expired` 后停止轮询、清除活跃任务，并显示"二维码已过期，请重新取码"。
[遗漏] `baseUrl` 会被持久化并用于后续请求，但未说明校验 → 保存和使用前限制为 HTTPS 且属于允许的微信 iLink 域名。
[遗漏] 非文本消息虽列为非目标，但没有定义收到后的行为 → 像飞书桥一样回复"不支持该类型"，不写入总控台，同时正常推进游标。
[遗漏] 长轮询只提到 `ret` 非零退避，未覆盖网络异常、超时、非法 JSON 和限流 → 使用有上限且带抖动的退避，并保证每轮异常不会退出整个循环。
[遗漏] 当前验收没有覆盖并发取码、重启恢复、停止中的长轮询和 `-14` 过期 → 为客户端生命周期补充模拟请求测试，并人工验证失败日志起止配对。
