# 弹框转微信选择 · Plan

## 大哥摘要

现在的毛病：总控台 AI（微信背后那个指挥台）干活时，时不时会弹一个"让你选一下"的框（要不要执行某命令、信不信任这个目录、第一次开权限全开的"是否接受"框等）。这种框在服务器上弹，你在微信里点不到，于是 AI 就一直卡在那等人点，微信也没回音。

这次要做：**只要总控台弹出这种选择框，就自动把框里的问题和选项推到你微信上，你直接回一个数字（或一句话），系统把你的选择敲回终端，AI 接着干**。做完后，微信端就能远程拍板，不用守在电脑前。

注意：这种"远程帮终端点选项"靠的是模拟敲键盘，绝大多数选择框能打通，但终端弹框样式五花八门，**不保证 100% 每种框都能完美驱动**——万一某个框敲了没反应，你发"取消"两个字就能解锁、换个说法重发。这点先跟你说清楚。

## 目标

- 总控台进入"等待输入"（弹框）状态且当前有微信在途请求时，自动把"问题 + 终端最后十几行（含选项原文）"推送到 owner 微信。
- owner 下一条微信消息被识别为"对弹框的回答"，原样敲进总控台 PTY（终端通道）+ 回车，而不是当成新指令。
- 验收（浏览器/微信可观察）：
  1. 在总控台 claude 里手动触发一个需要确认的操作（如让它跑一条会弹确认的命令）→ 微信收到一条"⚠️ 总控台在等你选择"开头、带选项文本的消息。
  2. 微信回 `1` → 终端对应选项被选中、AI 继续；LogsView 看到 `scope=wechat action=prompt-forward` 的起止配对，以及 `scope=wechat action=prompt-answer` 写入日志。
  3. 失败分支：微信回一个非法/敲了没反应的值，发"取消"能解锁（已有逃生指令）。
- 类型检查：`pnpm -F @aimon/server build` 通过。

## 非目标 (Non-Goals)

- 不做"把选项做成微信按钮卡片"——ilink 协议只发纯文本，回数字即可。
- 不替你消灭弹框本身（权限全开已尽力，剩下的框是 claude 内建、无法全关）。
- 飞书通道本轮不接这套（飞书没做单请求 pending 闸口，结构不同；先把微信打通，飞书另起）。
- 不做方向键逐项移动选择——统一用"敲数字/文本 + 回车"，键位映射越复杂越脆。

## 实施步骤

1. PendingRequest 增加 `awaitingChoice` 标记 + 记录已推送的弹框指纹（防重复推同一个框）。→ verify: build 通过
2. `onHubStatusChange`：`waiting_input` 且有 pending 时，不再走"孤儿自动解锁"，改为**判定是不是真弹框**（扫 PTY buffer 尾部找选项特征：`❯`/`1.`/`(y/n)`/`Do you`/`是否` 等），是→推微信 + 置 `awaitingChoice`；不是→维持原孤儿解锁逻辑。→ verify: 手动触发弹框，微信收到推送
3. `handleInbound`：进闸口前先判断 `awaitingChoice`——是→把本条消息当答案，`writeHubInput` 原样写入（不加 requestId 前缀），清 `awaitingChoice`，保留 pending 继续等 AI 最终回复。→ verify: 微信回数字后终端选项被选中
4. 推送/写入都包 `serverLog`（scope=wechat，action=prompt-forward / prompt-answer）起止配对。→ verify: LogsView 可见

## 边界情况

- 同一个弹框 Notification 可能连发多次 → 用弹框指纹去重，只推一次。
- owner 在 `awaitingChoice` 期间发"取消"→ 走已有逃生指令解锁，不当答案。
- 推送时 `pending.contextToken` 可能已过期 → 推送失败只记 error，不崩主流程（owner 可发"取消"重来）。
- claude 已经答完、只是停在主输入框（不是真弹框）→ 靠 PTY 特征判定排除，避免把"正常结束"误推成弹框。

## 风险与注意

- **TUI 交互天生脆**：敲数字+回车不保证驱动所有弹框样式；plan 已把"发取消解锁"作为兜底写进验收。
- 弹框检测是启发式（扫终端文本特征），可能漏判/误判；先覆盖 claude 最常见的几种框（权限确认、y/n、编号选择）。
- 运行中的部署在 `AIkanban-stable`，本修复落 `AIkanban-main`，需同步并重启服务才生效。
