# AI资讯雷达面板 · Plan

## 大哥摘要

这次会在左侧「外观」下面新增「AI资讯」按钮，点开后能看到最新 AI 资讯列表、数据更新时间和一键刷新。
点任一资讯，右侧会像阅读文章一样打开只读详情页，标题、入选理由和各来源链接都能直接查看。
刷新失败时不会清空已经看到的资讯；数据太旧、时间异常或暂时拉取失败时，页面会明确提示。
不会修改数据库或现有项目文件，只会新增入口和操作日志；打开资讯详情会顶掉你当前正在预览的那个文件页签（编辑区顶部的标签页，文件本身不受影响，再点一下就回来），切换项目后资讯详情仍保留。

## 目标

- 在浏览器左侧活动栏的「外观」下方能看到并点开「AI资讯」按钮；无论当前未选择项目、选择普通项目还是选择总控台项目，都能打开资讯侧栏。
- 资讯侧栏自动拉取 `daily-brief.json`，展示实际返回的全部条目，而不是假定固定 20 条；每条至少展示标题、重要度、来源数和时间。
- 侧栏顶部展示数据生成的绝对时间；`generated_at` 超过 36 小时、无效或来自未来时，显示清晰的陈旧或时间异常提示。
- 刷新按钮强制绕过 10 分钟后端缓存；刷新期间按钮不可重复点击。刷新失败时保留最后一次成功列表并显示错误，首次加载失败时显示错误空状态。
- 在浏览器里点任一条目后，右侧编辑区打开只读 Markdown（排版后的文本）详情页；详情包含标题、分类、重要度、入选理由、时间范围、来源链接和原文链接，链接在新窗口打开。
- 同一资讯重复点击时只保留一个详情页签；页签保存内容快照，资讯列表刷新或上游条目消失后，已打开详情仍可阅读。
- 资讯详情属于全局页签，切换项目时不消失；同时沿用现有"单个文件预览页签"行为，打开资讯会替换当前文件或提交详情预览，反之亦然。
- 后端提供 `GET /api/radar/daily-brief`，统一处理 15 秒超时、10 分钟内存缓存、`?force=1` 绕缓存、上游结构校验和结构化错误。
- 上游返回字段缺失或格式变化时，页面不崩溃：缺失的 `items`、`sources`、`reasons` 按空数组处理；整体结构不可用时后端返回明确错误。
- LogsView（应用内日志页面）能看到 `scope=radar`、`action=fetch` 的开始、成功或失败配对；后端日志包含 `force`、`cached`、`itemCount`、`generatedAt`。
- 验收失败分支：先成功加载资讯，再断开机器外网但保持本地服务运行并点刷新；浏览器保留旧列表、显示错误，LogsView 出现 `scope=radar` 的 ERROR。
- 通过可控的后端验证脚本模拟缓存命中、强制刷新、上游超时和错误结构，避免依赖真实上游故障才能验证。
- 完成交付前执行并通过：
  - `pnpm -F @aimon/web build`
  - `pnpm -F @aimon/server build`
  - `pnpm -F @aimon/server exec tsx scripts/radar-test.ts`

## 非目标

- 不自建或运行资讯抓取流水线，不 fork `ai-news-radar`，不运行其 Python 任务；数据继续来自 LearnPrompt 的公开 JSON。
- 不做订阅源管理、OPML 导入导出、已读状态、收藏、搜索、推送通知或历史归档。
- 不拉取约 2MB 的 `latest-24h.json`，本期只使用 `daily-brief.json`。
- 不持久化资讯列表和资讯详情；刷新浏览器后详情页签随现有 `openFiles` 一起消失。
- 不修改现有 `FilePreview`、`MarkdownView` 的通用行为，不为单一接口新增查询库或一次性服务层。
- 不修复技能市场"多 skill 仓库错抓第一个 SKILL.md"的已知问题。

## 实施步骤

1. 新增独立的资讯后端路由并注册到服务入口。
   - 新建 `packages/server/src/routes/radar.ts`，将硬编码上游地址、15 秒超时、10 分钟内存缓存、`force=1` 绕缓存、响应结构校验和错误映射集中在该路由内；不新增单独服务层。
   - 服务端响应类型放在路由附近，不新建仓库中不存在的 `packages/server/src/types.ts`。
   - 对上游数据做归一化：数组字段缺失时回落为空数组；无效条目允许侧栏安全跳过；整体结构错误返回明确的 502 类错误。
   - 使用 `serverLog` 记录 `fetch` 开始、成功和失败，日志附带 `force`、`cached`、条目数和生成时间。
   - 修改 `packages/server/src/index.ts` 注册 `/api/radar/*` 路由。
   - 如何验证：运行 `pnpm -F @aimon/server build`，并请求普通与 `?force=1` 两种地址，确认返回结构和缓存行为。

2. 增加可控的后端验证脚本。
   - 新建 `packages/server/scripts/radar-test.ts`，在不依赖真实上游故障的情况下验证正常响应、缓存命中、强制刷新、超时、非成功状态和错误结构。
   - 如何验证：运行 `pnpm -F @aimon/server exec tsx scripts/radar-test.ts`，全部场景通过。

3. 补齐前端数据契约和请求入口。
   - 在 `packages/web/src/types.ts` 增加前端使用的资讯响应、故事和来源类型。
   - 在 `packages/web/src/api.ts` 增加 `getRadarDailyBrief(force?)`，所有外部数据仍通过本地后端代理，不从浏览器直连 GitHub Pages。
   - 如何验证：运行 `pnpm -F @aimon/web build`，确认类型和请求调用通过编译。

4. 扩展活动栏与全局资讯页签模型。
   - 在 `packages/web/src/store.ts` 的 Activity 枚举中加入 `radar`；现有 localStorage（浏览器本地记录）清洗逻辑会自动认可新值，无需迁移。
   - 为 `EditorTabKind` 增加 `radar`，页签保存 `radarStoryId`、显示标题和 Markdown 内容快照；`radarStoryId` 必须参与唯一键，避免重复或错误去重。
   - 修改项目切换时的页签过滤逻辑：普通文件页签继续按现有规则处理，`kind='radar'` 的全局资讯页签不因切换项目被删除。
   - 不把资讯列表放进全局 store（共享状态）；列表状态由 `RadarView` 自己维护，重新进入侧栏时由后端缓存提供快速响应。
   - 如何验证：同一条资讯重复点击不重复建页签；切换项目后资讯页签保留；打开普通文件仍会按现有单预览页签规则替换资讯页签。

5. 增加活动栏入口和资讯侧栏。
   - 在 `packages/web/src/components/layout/ActivityBar.tsx` 的「外观」后加入「AI资讯」按钮，按钮始终显示。
   - 在 `packages/web/src/components/layout/PrimarySidebar.tsx` 增加标题和 `RadarView` 分发。
   - 新建 `packages/web/src/components/sidebar/RadarView.tsx`：负责首次加载、手动刷新、加载中、空列表、错误提示、旧数据保留、生成时间和资讯列表。
   - 首次加载与手动刷新使用 `logAction('radar', 'fetch', ...)` 记录起止；刷新中禁用按钮。
   - 列表支持任意条目数；重复或缺失 `story_id` 时使用故事标识、有效链接和位置组合出安全的列表键。
   - 如何验证：在浏览器分别处于未选项目、普通项目、总控台项目时打开侧栏；确认按钮位置、列表、空状态、刷新禁用和时间提示正确。

6. 在编辑区渲染只读资讯详情。
   - 在 `packages/web/src/components/editor/EditorArea.tsx` 为 `kind='radar'` 增加单独的只读 Markdown 渲染分支，不修改 `FilePreview`。
   - 打开详情前把故事转换为 Markdown 内容快照；转义标题、理由、来源名等上游文本中的 Markdown 特殊字符，避免内容改变页面结构。
   - 原文链接按 `primary_url` 优先、`url` 回落；来源和原文只渲染有效的 `http/https` 链接。现有 `MarkdownView` 已负责新窗口打开链接。
   - 缺少分类、理由、时间或来源时显示合理缺省文案，不因单个字段缺失导致详情页失败。
   - 如何验证：点击条目后检查页签标题、详情字段和外链；刷新列表或切换项目后，已打开详情仍可阅读。

### 给大哥的手动验收清单（交付后你自己点，不用懂代码）

- 左侧竖条「外观」🎨 下面出现「AI资讯」按钮，点开能看到资讯列表和顶部的数据时间；没选项目、选普通项目、选总控台时都能用。
- 点任一条资讯，右侧打开一篇排版好的详情页：标题、重要度、入选理由、各来源链接都在，链接点了在新窗口打开。
- 点"刷新"，刷新过程中按钮按不动；刷完时间更新。
- 断网再点刷新：旧列表还在、出现错误提示；左侧「日志」面板里能看到 radar 的红色 ERROR 条目。恢复网络再刷新即恢复。
- 切换到别的项目，刚才打开的资讯详情页签还在。

## 边界情况

- 上游返回 0 条、少于或多于约 20 条资讯时，侧栏按实际数量展示；0 条是正常空状态，不误报为系统错误。
- `story_id` 缺失或重复时，列表和页签仍有可用的唯一键；重复点击同一有效故事仍只打开一个详情页签。
- `sources`、`reasons`、`items` 缺失或不是数组时按空数组处理；单个故事严重损坏时跳过该故事，整体结构损坏时返回明确错误。
- `primary_url` 缺失时回落到 `url`；链接为空、格式错误或不是 `http/https` 时不渲染为可点击链接。
- 标题、理由或来源名含 Markdown 特殊字符时按普通文字显示，不生成额外标题、图片或链接。
- `generated_at` 超过 36 小时显示陈旧提示；无效或来自未来显示时间异常；同时显示绝对时间，避免只看相对时间产生误解。
- 首次加载失败时显示错误空状态；已有列表后的刷新失败只显示错误并保留旧列表。
- 刷新过程中连续点击不会发出重复请求；多个应用实例仍可能各自发请求，后端缓存负责降低上游压力。
- 未选择项目、选择普通项目、选择总控台项目时，资讯入口和侧栏都可用。
- 资讯详情切换项目后保留；普通文件页签仍维持现有项目切换规则，需防止修改过滤逻辑后误保留普通文件。
- 打开资讯会替换当前文件或提交详情预览，这是现有单预览页签语义；打开普通文件也会替换当前资讯详情。
- 刷新浏览器后资讯列表和详情页签不保留，符合本期不持久化的约定。
- 上游请求超时、网络不可达、返回非成功状态或返回非 JSON 时，后端返回结构化错误并记录 ERROR。

## 风险与注意

- 上游是公开 GitHub Pages 数据源，没有服务可用性承诺；保留后端代理可统一超时、缓存、结构校验和日志，前端直连虽然少一个路由但不采用。
- 资讯列表放在 `RadarView` 本地状态是最小改动；切离再进入侧栏会重新请求本地后端，但通常命中 10 分钟缓存，不引入新的全局查询状态。
- 全局资讯页签需要改动项目切换时的 `openFiles` 过滤逻辑，这是本任务最容易影响现有行为的部分；必须同时验证普通文件仍按原规则关闭、资讯页签单独保留。
- 页签保存 Markdown 内容快照而不是只保存 `radarStoryId`，会占用少量前端内存，但单次仅约一篇资讯且不持久化，可接受，并能避免刷新后详情失效。
- 后端类型只能约束本仓库代码，不能保证上游真实数据；因此必须做运行时结构校验，不能只依赖 TypeScript（编译时类型检查）。
- 组装 Markdown 时，上游文本和链接都属于外部输入；必须转义文本并限制链接协议，避免异常内容改变详情页面结构。
- 新增 `radar` Activity 后现有 localStorage 清洗逻辑不会误删它，无需迁移；旧版本读取到该值时会按现有规则回落默认活动。
- 本功能不修改数据库或用户项目文件；可感知的数据写入只有操作日志，落盘到 `packages/server/data/logs/YYYY-MM-DD.log`。
- 预计涉及约 10 个文件：新增后端路由、后端验证脚本、`RadarView`，并修改服务入口、前端类型/API/store、活动栏、主侧栏和编辑区；不修改 `FilePreview`、`MarkdownView`。
- 方案遵循项目记忆：全机器级功能使用独立 `/api/radar/*`，小范围缓存使用内存 Map，不引入新查询库，用户触发的异步刷新使用 `logAction`。
- 方案遵循 `dev/ARCHITECTURE.md` 的 Fastify（后端路由框架）路由、操作日志起止配对和前后端类型镜像约定。
