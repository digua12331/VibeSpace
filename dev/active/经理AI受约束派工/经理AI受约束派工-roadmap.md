# 经理AI受约束派工 · 后续版本路线图

> 第一版(受约束派工助理)已落地,见同目录 plan/context/tasks。本文件规划第二版起的拓展。
> 每版分两段写:**【给大哥什么】**(白话能力,非程序员看这段决定要不要做)+ **【技术内容/前置/风险】**(给执行 AI)。

## 一条贯穿主线(先读)

Claude+Codex 评估的结论钉死了顺序:**这套东西值不值得继续加码,取决于"它到底有没有替大哥省时间"**,而不是经理 AI 多聪明。所以:

> **第二版必须先做"量化"**(省了多少派工、一次验收通过率、合并后返工率)。**这三个数没证明第一版真省时间之前,不要做第三版起的自主化**——否则就是 Codex 警告的"批量制造待办和烂摊子的按钮"。

决定性变量永远是:**任务能否被准确拆开 + 后端硬检测能否识破"假成功"**。每加一版都要回头问:这版有没有恶化这两点。

## 第二版 · 让经理更省手 + 看得见效果(下一步做这个)

### 【给大哥什么】
- 一个**「召唤经理」按钮**:点一下就起经理 AI、不用自己建带关键词的任务。
- 子任务面板上**每张卡标出"它会动什么"**(改文件/动数据库/删东西),你一眼看清风险。
- 一个**「经理战绩」小面板**:这次帮你省了几次手动派工、几个任务一次验收就过、几个返工了——用数字告诉你它到底香不香。
- **自动合并(可选,默认仍关)**:边界允许时,验证+危险检测都过的安全子任务,经理可自动合并,免去你逐个点;危险的、动数据的永远还是停下等你。

### 【技术内容 / 前置 / 风险】
- **召唤入口**:前端加按钮 → 调现有"起 claude 会话 + 绑 task + 注入 skill",任务名自动带触发词。低风险,纯 UI + 复用。
- **danger 可视化**:`SubtaskSpec.danger` 第一版已存在(仅未展示)。DocsView 子任务行加 chip(参 vibespace-ui-decorator 调色板)。注意 danger 是**提示不是授权**,UI 文案别让大哥误以为"没标 danger 就安全"。
- **指标面板**:需后端记录每次派工/合并/返工(子任务被 reject 后重派、merge 后又改)。新增轻量埋点表或复用操作日志聚合。前端一个小面板读 `GET /api/.../subtask-metrics`。**这是第二版的重点,别砍**。
- **自动合并**:把第一版 UI 灰显的"允许自动合并"接上后端。安全条件:`managerAllowAutoMerge` 开 **且** verify 过 **且** 危险检测无命中 **且** 无 merge 冲突。经理 skill 解锁"全 review-ready 后调 approve-all"。风险高(破坏现有"人工放行"模型),所以放第二版而非第一版,且默认关、需危险确认才开。
- **触发条件**:第一版端到端跑通(步骤 10)+ 大哥实际用过几次。

## 第三版 · 半自主循环(soul flow 的克制版)

### 【给大哥什么】
经理不再"派完就干等",而是会**自己盯、自己救场**:某个子任务卡住超时 → 经理自动诊断、重派或升级给你;全部干完 → 自动汇总(第一版已有雏形);中途发现拆错了 → **提议**改任务图(改不改仍你拍板)。你更省心,但所有闸口照旧。

### 【技术内容 / 前置 / 风险】
- 这是 roadmap 里 **s11 自主化** 的核心,也是 LingTai 那种"AI 空闲自反思"的克制移植。
- 需要一个**定时/事件唤醒经理会话再跑一遍**的机制——目前仓库**没有**(context.md 步骤 5 调研已确认)。要新建:`statusManager.claimIdle` + `hub.ts:dispatch-to-idle-session` 注入 + 一个 tick。
- **铁律不变**:自检循环只能"诊断/重派/提议",**不能自己合并、不能开危险边界、不能绕确认**。每次自主动作要可见(日志 + 通知大哥)。
- **风险**:这是最容易跑飞、烧 token 的一版。必须配**预算/熔断**(参 `task-budget.ts` 轮数/时长/停滞上限),经理空转或反复重派要自动停。
- **硬性前置(不可跳)**:第二版的"经理战绩"指标证明第一版确实省时间。数字不好看就**不做这版**,回去改拆活质量/验收方式。

## 第四版 · 经理居中转发,让团队会配合(2026-06-09 重定义,不照搬 LingTai 邮箱)

### 【给大哥什么】
子工干活遇到拿不准的地方,能**回头问经理**(而不是猜错了埋头干);经理能中途给在跑的子工**追加约束/纠偏**。整队 AI 像有主管的团队那样配合——但**消息都过经理这个交换机**,你始终只盯一个经理脑子,不丢可控性。

### 【为什么不照搬 LingTai 邮箱(决策记录)】
LingTai 是"一群平级 AI 自组织",所以用点对点邮箱。咱们是"一个经理 + 几个子工",结构不同:让**经理居中转发**就够了,而且**大半零件已造好**——子工"完成/卡住"靠带外信号 + STATUS.md(第一版)、后端"有进展"自动注入经理(第三版 manager-tick)、注入通道(dispatch-to-idle)都在。真正缺的只是"子工能反问一句",给现有信号加个"提问"类型即可,不是一整套邮箱。

### 【技术内容 / 前置 / 风险】
- N4.1 子工反问 + N4.2 经理追加约束:复用现有 job-signal 轮询 + manager-tick + dispatch-to-idle,可控小改,不需新协议、不赌模型肯不肯用。
- N4.3 点对点邮箱:**降级为最后手段**,只有确有"子工绕开经理直接对话"的需求才做,且先 spike。重新引入"模型肯不肯用 MCP"风险 + 丢中心可控性。

## 每版交付前都要过的自检(继承第一版铁律)

1. 危险动作仍由**后端按实际 diff 硬拦**,新功能没给它开后门。
2. 合并/危险/确认三个**闸口没被绕过**。
3. 新增的"用户可感知操作"都有 `logAction`/`serverLog` 起止配对(scope=manager)。
4. 这版有没有恶化"任务可拆性 / 假成功可检性"这两个决定性变量。
5. 资深工程师自检:有没有过度设计(不做没人要的灵活性/可配置)。
