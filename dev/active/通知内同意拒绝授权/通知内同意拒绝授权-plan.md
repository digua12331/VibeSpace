# 通知内同意拒绝授权 · plan

## 大哥摘要
现在 AI（Claude）跑到一半要你授权（比如"需要你允许它运行命令/改文件"）时，系统右下角会弹个通知，你只能点"打开会话"跳进去再手动选同意。这次给**授权类**的通知加上"✅ 同意 / ❌ 拒绝"两个按钮——你在通知上点一下就直接替你回答，不用再跳进软件。其它两类通知不变：AI 列一堆选项让你挑的，还是点"打开会话"跳进去选（那种选项程序拿不到，没法放进通知）；AI 只是把话说完等你下一句的，也还是点击跳转。不动任何数据。

## 目标
- Claude 授权类等待（通知文案含 "permission"）的系统通知：动作按钮换成 `✅ 同意 / ❌ 拒绝`。
- 点"同意" → 给该终端发回车（接受 Claude 授权弹窗默认高亮的 "Yes"）；点"拒绝" → 发 Esc（Claude 弹窗的"No / 取消"）。
- 点通知主体（非按钮）仍跳转到该会话。
- 非授权类通知（AI 多选项 / 对话结束等待）：按钮与行为保持现状（打开会话 / 忽略）。

### 验收标准（浏览器可观察）
1. 让 Claude 触发一次授权请求（如让它跑一条需要确认的命令），把 VibeSpace 标签切到后台 → 系统通知出现 `✅ 同意 / ❌ 拒绝` 两个按钮。
2. 点"✅ 同意" → 不跳窗口，回到 VibeSpace 看该终端，Claude 已按"同意"继续往下跑。
3. 点"❌ 拒绝" → 该终端里 Claude 进入"被拒绝/让你说明"的分支。
4. 普通"对话结束/等待"通知仍是"打开会话 / 忽略"，点击跳转照旧。
5. UI 日志面板能看到 `scope=session` 的 `从通知同意/拒绝授权` 记录。

## 非目标
- 不解析终端画面去搬 AskUserQuestion 那种动态多选项（脆，AI 改版即坏）。
- 不新增后端端点；标签页**整个关闭**时退化为打开窗口手动处理（罕见，通常只是失焦）。
- 不改 codex/gemini/shell 的通知（授权语义只来自 Claude 的 Notification hook）。

## 实施步骤
1. `notify.ts`：判定通知类型（detail 含 permission = 授权类），把 `kind` 传进 SW 的 postMessage。验证：类型检查过。
2. `sw.js`：授权类用 `同意/拒绝` 动作；notificationclick 里 approve/reject → postMessage `session-response` 给已开页面（不抢焦点），无页面则开窗兜底。验证：通知显示对应按钮。
3. `main.tsx`：监听 `session-response` → 映射按键（同意=`\r`，拒绝=`\x1b`）→ `aimonWS.sendInput` + clearNotify + 操作日志。验证：点同意/拒绝终端有反应、日志面板有条目。
4. `pnpm -F @aimon/web build` 通过。

## 边界情况
- 通知还在但用户已回到页面再点同意：页面已聚焦，sendInput 照常работ，无害。
- 点同意时该终端其实已不在 waiting_input：发个回车/Esc 基本无害（空回车 ≈ no-op，Esc 无害）。
- 旧版浏览器无 Service Worker：走 legacy Notification，不支持动作按钮 → 授权类退化为点击跳转（可接受）。

## 风险与注意
- **同意=Enter、拒绝=Esc 依赖 Claude 授权弹窗的现有交互**（默认高亮 Yes、Esc=No）。Claude 大改交互时需调这两个按键——已抽成单个常量并注释，改一处即可。
- 只对 `kind==='permission'` 生效，绝不影响普通通知；用文案 `permission` 关键字网关隔离。
