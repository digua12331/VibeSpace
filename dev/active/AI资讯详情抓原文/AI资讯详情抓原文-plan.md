## 大哥摘要

现在点开一条 AI 资讯，详情页主要只有标题、来源等简要信息；这次会让详情页自动去原文网站抓取正文，并把可阅读的正文文字显示出来，不做 AI 摘要。
做完后去「AI 资讯」侧栏点开一条资讯，可以先看到“正在抓取原文”，随后看到明显多于标题的正文；抓不到时会明确提示失败，原有标题、来源和链接仍会保留。
系统只会抓取当前资讯列表里已经认可的原文地址，并检查不会误访问本机或公司内网；不会信任页面传来的任意网址。
不会修改现有资讯数据、历史内容或资讯列表样式；旧资讯离开当前列表后可能无法重新抓取，但已打开页面里的现有内容快照仍能看。

## 目标

让 AI 资讯详情页在打开时，按 `storyId`（每条资讯的内部编号）向后端请求原文；后端只从当前 `daily-brief` 缓存（资讯列表的短期内存副本）中解析对应 `primaryUrl`（首选原文地址），安全抓取网页，提取并返回纯文本正文，不返回原始 HTML（网页源代码）。

可验证的验收标准：

- 浏览器可观察行为：打开「AI 资讯」侧栏，点击一条可抓取资讯，详情页先显示“正在抓取原文”，数秒内显示明显多于标题的正文纯文本；重新打开同一条时可复用内存缓存，页面行为一致。
- 浏览器失败行为：通过断网、测试用非法故事编号或选择无法抓取的站点触发失败，详情页显示可重试的失败提示，同时保留原有 `radarMarkdown`（打开资讯时保存的详情内容快照）中的标题、来源和原文链接。
- 授权与安全：接口只接收 `storyId`，不接收客户端 URL；故事不在当前 `daily-brief` 缓存时拒绝抓取。初始地址和每次重定向（网站把请求带到另一个地址）都必须检查为公网地址，并把检查结果绑定到实际建立连接的 IP，拒绝本机、内网、链路本地地址和 DNS 重绑定（域名检查后临时改指向内网）的尝试。
- 抓取边界：用流式读取按实际收到的字节数在 2MB 处中止，不能只信 `Content-Length`（网站声明的响应大小）；限制重定向跳数、请求耗时和提取后的正文字符数；非 HTML、空壳页、登录页或正文低于最短长度时按失败处理。
- 提取质量：使用 `@mozilla/readability`（成熟的网页正文提取器）和 `linkedom`（把网页源代码解析成可读取结构的轻量工具）提取正文；自测覆盖真实世界脏 HTML（结构不规范的网页源代码）固件，并确认返回值不含原始 HTML 标签。
- 日志验收：在 LogsView（应用内日志页面）看到 `scope=radar action=fetch-article` 的开始、成功或失败起止配对；人工触发一次失败并看到 ERROR。后端落盘日志 `packages/server/data/logs/YYYY-MM-DD.log` 只记录域名、耗时、结果和经过清理的失败原因，不记录完整 URL、正文或 HTML。
- 类型检查与构建：`pnpm -F @aimon/server build` 和 `pnpm -F @aimon/web build` 均通过。
- 自测脚本：新增并通过 `pnpm -F @aimon/server exec tsx scripts/radar-article-test.ts`，覆盖正常提取、脏 HTML、未知 `storyId`、私网地址、DNS 重绑定、重定向到私网、重定向过多、实际流式响应超过 2MB、非 HTML、正文过短和正文字符数截断。

## 非目标

- 不做 AI 摘要、翻译、动态网页渲染，也不自动改抓同一故事的其他来源。
- 不改变资讯列表样式，不改上游 ai-news-radar 数据，不把抓到的正文持久化到磁盘或数据库。
- 不保证所有网站都能抓取；需要登录、验证码、JavaScript（网页脚本）动态渲染或主动拦截程序访问的网站允许失败。

## 实施步骤

1. 明确安全抓取边界与可测试接口：在现有 `/api/radar/*` 能力内设计 `GET /api/radar/article?storyId=...`，由后端从当前 `daily-brief` 缓存解析唯一 `primaryUrl`；扩展现有测试注入接口，使其能模拟状态码、响应头、流式响应体、重定向和实际连接地址。先核实当前 Node/undici（Node 网络请求底层库）的 dispatcher/lookup 接口能否把公网 IP 校验绑定到实际连接；若不能保证，则不得用仅“请求前解析一次域名”的不安全方案交付。如何验证：用自测证明客户端传 URL 无入口、未知故事编号被拒绝、连接到私网 IP 和 DNS 重绑定均被拒绝。
2. 实现受控网页抓取：初始 URL 必须由 `storyId` 命中缓存白名单获得，且缓存 URL 本身仍需通过公网地址检查；设置 `redirect: 'manual'`（由程序逐跳处理跳转），限制最大跳数，每一跳重新校验地址并检查实际连接 IP；设置超时、常规浏览器标识、HTML 类型检查，并按流式实际字节数在 2MB 处中止。如何验证：自测覆盖正常响应、跳转成功、跳转到私网、跳转过多、伪造较小 `Content-Length` 但实际超限、超时和非 HTML 响应。
3. 提取并约束正文：新增 `@mozilla/readability` 与 `linkedom` 依赖，把抓到的 HTML 解析后只返回纯文本正文及必要的安全元数据；用最短正文长度识别动态空壳、登录页或提取失败，用最大正文字符数避免超长内容拖慢 MarkdownView（详情页正文显示组件），并增加真实脏 HTML 固件测试。如何验证：自测确认常见新闻页和脏 HTML 能提取正文，空壳/登录页按失败处理，超长正文被截断，响应中不含脚本、标签或原始 HTML。
4. 接入后端路由、缓存与隐私日志：沿用现有 `RadarError`（统一错误格式）和短期内存缓存，按已授权的原文地址缓存成功提取结果，避免重复点击反复抓取；使用 `serverLog` 记录抓取开始、成功耗时和失败 ERROR，日志只保留域名及必要结果，不保留完整 URL、正文、HTML，也不让底层错误信息泄露完整 URL。如何验证：连续请求同一故事确认复用缓存；检查 LogsView 和当天 JSONL（每行一条日志记录）日志均有起止配对且不存在完整 URL 或正文；人工触发一次失败确认 ERROR。
5. 改造资讯详情页异步加载：新增独立的 `RadarStoryView`（资讯详情显示组件），由现有 radar tab 的 `storyId` 调用新接口；显示加载中、正文成功、失败可重试三种状态，始终把现有 `radarMarkdown` 快照作为兜底。前端请求使用 `logAction('radar', 'fetch-article', ...)` 包装，补齐 API 客户端和返回类型。如何验证：浏览器点击可抓取资讯看到加载中再出现正文；点击已离开当前缓存的旧 tab 或主动制造失败时，看到失败提示且原有标题、来源、链接不消失；重试可再次发起抓取；LogsView 有 `scope=radar action=fetch-article` 起止配对和失败 ERROR。
6. 完成全链路验证与范围核对：运行后端自测、后端构建和前端构建，检查本次改动只涉及计划内的雷达路由、正文提取/测试、前端详情/API/类型和依赖清单；手工检查日志隐私和失败兜底。如何验证：`pnpm -F @aimon/server exec tsx scripts/radar-article-test.ts`、`pnpm -F @aimon/server build`、`pnpm -F @aimon/web build` 全部通过，并按目标中的浏览器与日志验收步骤完成一次成功和一次失败操作。

## 边界情况

- 当前 `daily-brief` 缓存中找不到 `storyId`：不接受 tab 中的旧 URL 作为替代，也不抓任意地址；显示抓取失败，继续展示 tab 内已有 `radarMarkdown` 快照。
- 缓存命中的 `primaryUrl` 指向本机、内网、链路本地地址，或域名解析结果包含/切换到这些地址：拒绝请求；公网检查必须覆盖实际连接 IP，而不只是字符串和请求前 DNS 结果。
- 原文发生多次重定向：逐跳处理并逐跳重新授权公网目标，超过最大跳数或任意一跳不安全即失败。
- 响应没有 `Content-Length`、声明大小不准或分块持续传输：按实际读取字节累计，超过 2MB 立即中止。
- 返回 PDF、图片、JSON、验证码页、登录页、JavaScript 动态空壳、乱码或过短正文：按不可提取处理，不把垃圾内容显示成正文。
- 正文极长：提取后按设定的最大字符数截断并给出截断标记，防止详情页渲染卡顿。
- 同一故事有多个来源：首版只抓 `primaryUrl`，失败不自动尝试其他来源，避免扩大请求范围和用户等待时间。
- 外部站点暂时故障、限流或拒绝程序访问：显示可重试失败提示；不影响资讯列表和已有详情快照。

## 风险与注意

- 最大风险是 SSRF（让服务器被诱导访问本机或内网）。仅凭缓存白名单仍不够，因为缓存中的外部 URL 也可能解析到私网；必须同时做到 `storyId` 授权、逐跳公网检查和实际连接 IP 绑定。实施前要核实当前 Node/undici 的连接钩子能力，无法证明安全就停止该抓取方案，而不是降级为仅请求前 DNS 检查。
- `@mozilla/readability` 对 GitHub、论坛、Hacker News 评论页等非文章页面的提取质量可能不稳定；首版以“正文达到最短长度且为纯文本”为成功线，不引入自写启发式提取器或 AI 修复，避免结果不可预测。
- 新依赖会扩大服务端安装体积，但相较自写网页正文规则，成熟提取器对真实脏 HTML 更可靠；本轮采纳 `@mozilla/readability + linkedom`，放弃自写启发式方案。
- 旧 tab 没有原文 URL 是既有事实；接口改为只接收 `storyId` 后，离开当前缓存的旧资讯不能重新抓取，这是为了安全接受的限制，已有 `radarMarkdown` 快照负责兜底。
- 抓取属于用户可感知操作，必须有前后端起止日志和失败 ERROR；日志中的底层异常也要清理，避免完整 URL、正文或 HTML 经错误堆栈落盘。
- 项目记忆已对齐（auto.md 显式引用：「接收外部仓库地址并交给子进程前，必须先白名单解析再重组安全 URL」「前端新增用户主动触发的执行类动作时，用 logAction 包住异步主流程」「全机器级能力优先挂独立 /api/<feature>/* 路由」）；manual.md 偏好已对齐：交付前不跑 browser-use 自动验收，浏览器验收项由大哥手动验，handoff 给「点哪里看」指引。dev/ARCHITECTURE.md 扫过无 radar 相关章节（该面板是近期新增）。项目架构沿用 `packages/server` 后端、`packages/web` 前端和现有 `/api/radar/*` 边界，不新建平行能力。

> [Codex 综合主笔] 全部采纳安全授权、逐跳连接校验、成熟正文提取、脏 HTML 测试、旧 tab 快照兜底和日志隐私要求；放弃客户端传 URL、自写启发式提取、自动尝试其他来源与动态渲染，因为它们会扩大安全面、实现范围或导致结果不可控。
