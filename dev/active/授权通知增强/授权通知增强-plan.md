# 授权通知增强 · plan（Claude 草案，待 Codex 会审）

## 大哥摘要

这次改 Claude 找你要授权时弹出来的那个系统通知（桌面右下角的小弹窗），四件事：

1. **点"同意"= 以后不再问**：现在弹窗里的"✅ 同意"只放行这一次，下次同类命令还会再弹。改成点一下相当于选 Claude 那个框里的第 2 项"同意且以后不再问"，省得反复弹。按钮文字改成"✅ 同意并不再问"。
2. **通知里信息多一点**：弹窗里多显示是哪个项目、哪个 AI（claude/codex 等）、Claude 那句权限说明。（说明：Claude 只把"需要用 Bash 的权限"这种**一句话**发给我们，那条具体命令原文它没给，所以通知里显示不了 `Invoke-WebRequest …` 那一整行原文——除非再多接一道后端管线，这点下面留给你定。）
3. **看别的项目时也会弹**：现在只要 VibeSpace 窗口在最前面就不弹通知。改成——只有当你**正盯着发通知的那个项目**时才不弹；你在看别的项目、或在"全部 sessions"列表时，照样弹（你要的就是这个）。
4. **项目列表上点红点**：左边项目列表里，哪个项目有 AI 在等你授权，就在那条项目上点一个小红点；点进那个项目后红点消失。

不会动到你现有的任何数据、项目、会话；纯前端通知行为调整，做错了刷新页面就回来。

## 目标

改完后可验收的标准：

- **路 B（验收点）**：Claude 弹授权框时，桌面通知上是"✅ 同意并不再问 / ❌ 拒绝"两个按钮；点"同意并不再问"后，会话里那个授权框选中的是**第 2 项**（"Yes, and don't ask again"），不是第 1 项。
- **信息更多（验收点）**：通知标题含项目名，正文含 AI 名 + Claude 的权限提示文字。
- **跨项目（验收点）**：当前选中项目 A、窗口在前台时，项目 B 的会话进入等待授权 → 桌面通知**会弹**；而项目 A 自己的会话等待授权时（你正看着 A）→ **不弹**（你已经在现场）。
- **项目列表标记（验收点）**：项目 B 有会话等授权时，左侧项目列表里 B 这一行出现红点；点选 B 后红点消失。同时"激活/未激活"分页的"激活"标签上也能看到有未处理提醒（因为等授权的会话一定在激活分页里，避免你停在另一个分页看不到）。
- **构建门槛**：`pnpm -F @aimon/web build` 通过（含 TypeScript 类型检查）。

## 非目标（本次不做）

- 不改 Claude 授权框本身的交互（那是 Claude Code 的，不是我们能改的）。
- 不做"通知里显示完整命令行原文"——除非你在下面的分叉里选了"要"，否则本轮只显示 Claude 给的那句提示。
- 不动后端 SQLite 表结构、不动会话/项目数据。
- 不做通知偏好的设置开关（你明确选了路 B，不再额外加"只这次同意"的可配置项）。

## 需要你拍板的分叉（有用户感知差异）

**分叉①：在"全部 sessions"列表时算不算"在现场"？**
- 方案默认：把"全部 sessions"视图当作"没在盯具体项目"→ 照弹通知。
- 备选：在"全部 sessions"里也算在现场 → 不弹。
- 我建议默认（照弹），因为你要的就是"不在那个项目界面就提醒我"。

**分叉②：通知要不要显示具体命令行原文？**
- 默认：不显示（Claude 没把命令发给我们，要显示得在后端多存一份"上一条工具调用"再拼进通知，多动一个后端文件、有少量出错风险）。
- 备选：要 → 我加这道后端管线，通知正文里就能看到 `Invoke-WebRequest …` 这种原文。
- 我建议先默认（不显示），需要再加。

## 实施步骤

1. **路 B 改键 + 改文案**
   - `packages/web/public/sw.js`：`PERMISSION_ACTIONS` 里 approve 的 title 改 "✅ 同意并不再问"。
   - `packages/web/src/main.tsx`：`NOTIFY_RESPONSE_KEYS.approve` 由 `'\r'` 改为 `'\x1b[B\r'`（先按一下方向下键把高亮从第 1 项移到第 2 项，再回车确认）。
   - 验证：实际在会话里触发一次 Claude 授权框，点通知"同意并不再问"，观察框里选中的是第 2 项。（这条依赖 Claude 框的现有布局，是单点常量，Claude 改版时改这一处即可。）

1b. **顺手修一个既有隐患：通知按键只投递给一个页面（Codex 会审发现）**
   - 现状 `packages/web/public/sw.js` 的 `notificationclick` 把 `session-response` `postMessage` 给**所有**打开的 VibeSpace 页面。若你开了两个标签页，点一次"同意"会给同一终端发两次按键——路 B 后这个 bug 更扎眼（双发方向键+回车可能误选第 3 项 No 或乱跳）。
   - 改：approve/reject 分支只挑**一个** client 投递（优先聚焦的那个，没有就取第一个），不再 `for` 全发。
   - 验证：开两个标签页，点一次通知"同意"，终端只收到一组按键、授权框只动一次。

2. **通知信息加料**
   - `packages/web/src/notify.ts`：组装 body 时把 agent + detail 拼进去（如 `${agent} · ${detail}`），title 维持含项目名。
   - `packages/web/public/sw.js`：照常透传 body。
   - 验证：弹窗正文能看到 AI 名 + 权限提示。

3. **跨项目弹（改抑制条件）**
   - `packages/web/src/store.ts` `updateSessionStatus`：计算 `suppress = isPageFocused() && selectedProjectId === sess.projectId`，传给 `notifyWaitingInput`。
   - `packages/web/src/notify.ts`：把内部直接调用 `isPageFocused()` 的早退，改成用传入的 `suppress` 参数判定（保留 `Notification` 支持与权限检查）。
   - 验证：选中 A、前台，B 等授权 → 弹；A 等授权 → 不弹。

4. **焦点回归时只清"当前项目"的提醒，别清掉别项目的红点**
   - `packages/web/src/store.ts`：window focus / visibilitychange 现在调 `clearAllNotify()`（清全部）。改成只清"当前选中项目"的提醒会话，别的项目红点保留。
   - 验证：在 A，B 有红点；点回窗口 → B 红点还在；切到 B → B 红点消失。

5. **项目列表红点**
   - `packages/web/src/store.ts`：`selectProject(id)` 时清掉该项目下所有 notifying 会话（点进项目即视为已读）。
   - `packages/web/src/components/layout/ProjectsColumn.tsx`：由 `notifyingSessions` + `sessions` 推出"哪些项目在催"，给项目行加红点；给"激活"分页标签加一个有提醒的标记。
   - 验证：见目标里的"项目列表标记"验收点。

6. **操作日志**
   - "从通知同意/拒绝授权"的起止/结果日志已存在（main.tsx `handleNotificationResponse`），本次保持；approve 语义变化在 meta 里能体现（response=approve）。无新增 mutation API，前端埋点沿用 `pushLog`。
   - 验证：点通知后 LogsView 出现 `scope=session msg=从通知同意授权` 条目。

7. **构建验收**
   - `pnpm -F @aimon/web build` 通过。

## 边界情况

- **方向键+回车依赖 Claude 框布局**：若 Claude 把"不再问"挪走或改成非第 2 项，approve 会选错项 → 单点常量，已注释标注。
- **非授权类通知（generic）**：保持"打开会话/忽略"，跨项目弹的新规则同样适用（你在别处时也会被提醒有会话等输入）。
- **等授权会话所属项目在"未激活"分页**：理论上等待授权的会话是活的，必在"激活"分页；仍给"激活"标签加标记，避免你停在"未激活"分页漏看。
- **多个项目同时催**：每个项目各自红点；标签标记只表达"有/无"，不堆数字（避免误读）。
- **窗口整个关闭点通知**：沿用现有退化逻辑（开窗手动处理），本次不动。

## 风险与注意

- 改抑制条件会让通知更频繁（这是你要的效果），但要确认不会在你正盯着的项目里冒出多余通知——靠 `selectedProjectId === sess.projectId` 守住。
- `clearAllNotify` 的语义从"全清"变"清当前项目"，要回查所有调用点（focus 回归、tab 可见）确认没有别处依赖"一回来全清"。
- 纯前端改动，无破坏性变更、不动数据；做错刷新即恢复。

## 多模型 Plan 会审

> [Codex 评审] 确认抑制判断应从 `notifyWaitingInput` 内部上移到 `store.updateSessionStatus`（通知函数本身拿不到"当前选中项目"）——与草案一致。
> [Codex 评审] 会话类型里 `projectId` 是必填，项目红点聚合可直接由 session 反查项目，**无需新增后端字段**——确认草案路径可行，不动后端。
> [Codex 评审] 输入只是把字符串原样写进 PTY，VibeSpace 这侧不会丢转义；`\x1b[B\r` 选第 2 项的真正风险只在"Claude 授权框是否始终接受方向键"，故列为单点常量 + 需人工验一次。
> [Codex 评审 · 新增点] **既有隐患**：Service Worker 当前把"同意/拒绝"广播给所有打开页面，双标签页会向同一终端发两次输入。已据此新增步骤 1b（只投递一个 client）。
> [Claude 白话化兜底] Codex 仍在跑长时分析（>14min，未出最终格式清单），但其捕获到的评审结论已全部折入本 plan（步骤 1b + 步骤 3 抑制上移 + 风险段方向键说明）；为不让外部基础设施拖慢交付，按工作流"失败/超时回退"条提前定稿。大哥摘要、术语翻译、manual.md 偏好（默认档双模型 / build 即门槛 / 不跑 browser-use）均已对齐。
