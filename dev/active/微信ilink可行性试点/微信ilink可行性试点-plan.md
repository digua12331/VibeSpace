# 微信ilink可行性试点 · Plan

## 大哥摘要

这次不改产品代码，只做一个试验：验证「个人微信能不能替代飞书」这件事在技术上走得通。我会写一个独立的小测试脚本（放在本任务目录里，跟 VibeSpace 主程序无关），你需要配合两步：**用手机微信扫一次码**（把你的微信跟测试机器人绑定）、**在微信里给机器人发一两句话**。整个过程不会动你现有的飞书接入、任何数据和界面。

试验要回答三个问题，其中第 3 个最关键：
1. 扫码登录能不能拿到凭证（token，机器人后续干活的"钥匙"）？
2. 你在微信发话，本机脚本能不能收到、能不能回复？
3. **没人发话时，机器人能不能主动给你推送一条消息**（这决定"任务干完了提醒你"这类功能在微信上能不能实现；不行的话飞书就不能删）。

## 目标与验收标准

- 验收 1（登录）：运行脚本后生成二维码图片，手机扫码确认后，脚本打印"登录成功"并把 token 存到本地文件（已加入 .gitignore，不会进代码仓库）。
- 验收 2（收/回消息）：大哥在微信里给机器人发"测试"，脚本终端打印出这条消息，且微信里收到脚本的自动回复。
- 验收 3（主动推送，关键题）：在大哥**不发任何新消息**的前提下，间隔若干分钟后运行推送命令，微信里能收到机器人主动发来的消息。分别测试"复用旧 context_token（对话关联令牌，来自之前那条入站消息）"和"不带令牌"两种发法，记录各自结果。
- 失败分支：故意用错误 token 调接口，脚本能打印出明确的错误信息而不是崩溃。

## 非目标

- 不改 VibeSpace 产品代码（不动 packages/ 下任何文件）。
- 不实现媒体消息（图片/语音/文件）、群聊、长期断线重连——那是迁移任务的事。
- 不在本任务里决定"路 1（cc-connect）还是路 2（自研替换）"，试点结论出来后再议。

## 实施步骤

1. 下载并留存 ilink 接口参考文档到任务目录（已完成）。→ verify: 文档文件存在。
2. 写试点脚本 `ilink-poc.mjs`（login / listen / push 三个子命令）。→ verify: `node --check` 语法通过。
3. 把 token 状态文件、二维码图片加入 .gitignore。→ verify: `git status` 不出现这两个文件。
4. 探测接口连通性（取一次二维码，不需要大哥操作）。→ verify: 接口返回合理 JSON。
5. 大哥扫码登录（需要大哥手机操作）。→ verify: 验收 1。
6. 收发消息测试（需要大哥在微信发话）。→ verify: 验收 2。
7. 主动推送测试（间隔后无触发推送）。→ verify: 验收 3，两种发法各记录一次结果。
8. 产出试点结论，写回本目录 `试点结论.md`，给出"微信能否替代飞书"的明确回答。→ verify: 结论文件存在且回答了三个问题。

## 边界情况

- `get_bot_qrcode` 可能要求 OpenClaw 平台注册/审核（参考文档第八节提到此未知项）——若首个接口就被拒，试点直接得出"当前无法独立接入"的结论，转评估 cc-connect 路线是否同样被挡。
- 二维码有有效期，扫码步骤需要大哥在场时才执行。
- token 有效期未知：若中途失效，记录失效时长，这本身就是试点要的数据。
- 接口可能有速率限制，脚本轮询间隔保守设置。

## 风险与注意

- token 等于"以你微信身份收发消息"的钥匙，只存本地且不进 git。
- 腾讯条款允许其随时变更/终止该服务，试点结论需注明这一前提。
- 假设：Node 22 自带 fetch 可直连 `ilinkai.weixin.qq.com`（无需代理）。若网络不通，先排查本机网络再下结论。
