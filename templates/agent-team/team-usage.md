# 团队使用说明书（team-* 通用开发团队）

> 本文件随工作流装配进项目（`.aimon/docs/team-usage.md`），写给**主 AI** 看：项目里有一支四人通用团队（`.claude/agents/` 下 `team-*` 文件），什么活派给谁、什么活别派。主理人不用读这份——团队对他透明，他只看交付。

## 团队名册

| 角色 | subagent_type | 工具 | 派工场景 |
|---|---|---|---|
| 调研员 | `team-explorer` | 只读 | plan/context 阶段摸代码、画地图、并行多目录调研 |
| 实施工 | `team-implementer` | 可写可跑命令 | tasks 阶段接"文件白名单 + 做什么 + 怎么验证"齐全的明确切片 |
| 验收员 | `team-verifier` | 只读+跑命令 | 对照 verify 清单逐条跑项目自己的检查命令，报 PASS/FAIL/SKIP |
| 审稿员 | `team-rules-auditor` | 只读 | 交付/提交前审改动合不合项目规矩，出 BLOCK/WARN/INFO |

**所有成员上岗自带"大脑加载协议"**：干活前先读本项目的 `CLAUDE.md`、`dev/memory/`、`dev/ARCHITECTURE.md`、`.aimon/skills/` 相关文件，回复第一行声明读了什么、缺了什么。**收到没有 `[大脑]` 声明行的返回，视为不合格，重派。**

## 派工守则

1. **派工三件套**：给 team-implementer 的派工必须含**文件白名单、做什么、怎么验证**；给 team-verifier 的必须是**逐条 verify 清单**。缺件它们会退回——这是设计，不是故障。
2. **明确返回约束**：派任何角色都写明"返回 ≤N 行、什么格式"，否则子工的长篇输出会吃掉你的上下文。
3. **并行用在独立的活上**：多目录调研可以同时派几个 explorer；互相依赖的改动不要并行派 implementer。
4. **链式协作的标准节奏**：explorer 摸底 → 你写 plan/tasks → implementer 干切片 → verifier 对 verify 清单 → rules-auditor 审稿 → 你汇总交付。每一环的产出喂给下一环，**决策环节（plan、取舍、跟主理人对话）永远是你自己**。

## 派工模板（照填即合格）

派实施工（缺任何一栏它会退回）：

```
任务：<一句话说做什么>
文件白名单：<只允许改的文件，逐个列>
做法要点：<2-5 行；引用 plan/tasks 的对应条目>
验证：<完成后要跑什么命令 / 检查什么>
返回：≤15 行，按你的"完工动作"格式
```

派验收员：

```
验收清单：
V1 <可观察的验收项> 
V2 <...>
提示：项目的检查命令优先从 <CLAUDE.md / README / 构建配置> 找
返回：每条一行 PASS/FAIL/SKIP + 汇总行
```

## 什么活别派（戒律）

- **规划与决策**：plan/context 的设计判断不外包——子工没法跟主理人对话，决策外包是反模式。
- **排障 debug**：需要连续追踪假设链，派出去就断链，自己干。
- **流式连改**：后一步依赖前一步结果的连续编辑，主线性做比派工快。
- **单个简单搜索**：一次 Grep 能答的别派 explorer。
- **"感觉派子工比较高级"**：这是过度工程的红灯。

## 故障处理

- 子工返回"派工不完整/不明确"：补齐三件套重派，不要换个说法硬塞。
- implementer 报熔断（2 次验证不过）：你接手或升级给主理人，不要原样重派第三次。
- **子工心跳**：派出去的子工长时间没有任何产出（远超同类活的正常耗时），按"疑似卡死"处理——多半卡在你看不见的确认弹窗或外网请求上。主动终止它，捞回已有的部分结果，把剩余工作换成你自己前台做或换一条不依赖该工具的路线；**不要原地干等，也不要原样重派**。
- 角色文件被项目自定义过（内容改了）：以项目里的版本为准——它是主理人或前任主 AI 的本地化成果。

## 团队的进化（模板回写）

干活中发现某个团队成员的毛病属于**通用缺陷**（换任何项目都会犯，比如返回格式约束不够、缺某条纪律），而不是本项目特有问题时：

- **不要只在本项目里改它的文件**——那只治了一个项目。
- 在交付摘要里向主理人上报一句："<角色> 的 <问题> 是母版级的，建议回 VibeSpace 仓库改 `templates/agent-team/` 母版，再到各项目重新应用工作流即可自动升级（你没本地化改过的文件才会被刷新）。"
- 本项目特有的问题才就地改本项目的角色文件（改完它就不再跟随母版升级，这是预期代价）。
